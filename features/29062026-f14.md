# 29062026-f14 — Limpieza de R2

> Sub-plan de la **Fase 14** del fundational plan (`29062026-fundation-plan.md`).
> Implementa dos mecanismos complementarios de limpieza de imágenes en R2: eliminación explícita después de cada análisis exitoso, y una lifecycle rule de R2 como red de seguridad para objetos huérfanos (uploads sin análisis, fallos de eliminación).

- **Fase**: F14 - Limpieza de R2
- **Fecha**: 29-06-2026
- **Estado**: cerrada (limitación de alcance: solo eliminación inmediata vía `deleteR2Object`; la lifecycle rule de R2 se omite por decisión del usuario, que prefiere gestionarla luego desde un panel admin)
- **Prerequisitos**: F11 (uploads en R2 con prefijo `uploads/YYYYMMDD/`), F13 (endpoint `/api/analyze` donde se integra la eliminación)
- **Bloquea**: nada directamente, pero sin F14 los objetos en R2 crecen indefinidamente

---

## Objetivo

Las imágenes de los usuarios son datos sensibles — deben existir en R2 el mínimo tiempo posible. La estrategia es de doble capa:

1. **Eliminación inmediata** (happy path): el endpoint `/api/analyze` elimina el objeto de R2 después de retornar los resultados al cliente. El pipeline de análisis ya terminó, la imagen ya no sirve.

2. **Lifecycle rule** (red de seguridad): regla en el bucket de R2 que auto-elimina cualquier objeto en el prefijo `uploads/` después de 24 horas. Cubre los casos en que la eliminación explícita no ocurrió: upload sin análisis, fallos de red, crashes del serverless.

---

## Decisiones ya tomadas

| Decisión | Valor |
|---|---|
| Momento de eliminación explícita | Después de `runAnalysisPipeline` exitoso, antes de `return ok(results)` |
| `deleteR2Object` ante fallo | Fail-open: loguea el error, no lanza. El lifecycle rule limpiará |
| Lifecycle rule TTL | 24 horas después de creación del objeto |
| Prefijo del lifecycle rule | `uploads/` — cubre toda la estructura `uploads/YYYYMMDD/<uuid>.<ext>` |
| Configuración del lifecycle | Script TypeScript ejecutable una vez (`scripts/setup-r2-lifecycle.ts`) — reproducible y versionado |
| Eliminación en path de error | No se elimina si el pipeline falla — el usuario puede reintentar subiendo de nuevo. El lifecycle rule elimina el objeto huérfano dentro de 24h |

## Cierre — Definition of Done

- Después de un análisis exitoso, el objeto en R2 ya no existe (verificado en R2 Dashboard o con `HeadObjectCommand`).
- Si `deleteR2Object` falla (simular con una key inválida): el endpoint sigue retornando 200 con los resultados. El error aparece en logs.
- Ejecutar `pnpm setup:r2-lifecycle` configura la lifecycle rule en el bucket. Verificar en el R2 Dashboard que la regla existe con TTL de 1 día y prefijo `uploads/`.
- Objetos creados con más de 24 horas en el prefijo `uploads/` son eliminados automáticamente por R2 (verificación en F19 con objetos de prueba).
- `pnpm build` sin errores TypeScript.

### T14.1 — `src/lib/r2Delete.ts` — Helper de eliminación

```ts
import { DeleteObjectCommand } from '@aws-sdk/client-s3'
import { r2 } from '@/lib/r2'
import { env } from '@/lib/env'

export async function deleteR2Object(objectKey: string): Promise<void> {
  try {
    await r2.send(new DeleteObjectCommand({
      Bucket: env.r2.bucketName,
      Key:    objectKey,
    }))
  } catch (e) {
    console.error('[r2] deleteR2Object failed for key:', objectKey, e)
    // Fail-open: el lifecycle rule de R2 limpiará el objeto dentro de 24h
  }
}
```

**Por qué fail-open y no lanzar si la eliminación falla**: el análisis ya se completó y los resultados están listos para el usuario. Fallar el request por una eliminación fallida es una mala experiencia — el usuario no recibe sus resultados por un error de infraestructura secundario. El lifecycle rule es la garantía real de limpieza.

**Por qué la eliminación es síncrona** (se espera con `await`): en Vercel Node.js serverless, el proceso termina cuando la función retorna la `Response`. No hay forma de ejecutar código después de enviar la respuesta (a diferencia de Edge Runtime con `waitUntil`). La eliminación síncrona agrega ~50-150ms de latencia después del pipeline de IA, que ya duró ~8s. El impacto es mínimo.

**`DeleteObjectCommand` en R2**: R2 es compatible con la API de S3. `DeleteObjectCommand` funciona igual. Si el objeto no existe (ya fue eliminado por otra vía), R2 retorna éxito igualmente — idempotente por diseño.

**Criterio de aceptación**: llamar `deleteR2Object('uploads/YYYYMMDD/<uuid>.jpg')` con una key existente elimina el objeto. Llamarlo con una key inexistente no lanza error. Llamarlo con el cliente R2 apuntando a un endpoint inaccesible loguea el error y retorna sin lanzar.

### T14.2 — `scripts/setup-r2-lifecycle.ts` — Script de configuración del lifecycle

Script ejecutado **una sola vez** para configurar la lifecycle rule en el bucket. Se ejecuta manualmente o en el setup inicial del proyecto.

```ts
import { PutBucketLifecycleConfigurationCommand } from '@aws-sdk/client-s3'
import { r2 } from '../src/lib/r2'
import { env } from '../src/lib/env'

async function setupLifecycle(): Promise<void> {
  await r2.send(new PutBucketLifecycleConfigurationCommand({
    Bucket: env.r2.bucketName,
    LifecycleConfiguration: {
      Rules: [
        {
          ID:     'cleanup-uploads',
          Status: 'Enabled',
          Filter: { Prefix: 'uploads/' },
          Expiration: { Days: 1 },
        },
      ],
    },
  }))
  console.log('[setup] R2 lifecycle rule configured: uploads/ → expire after 1 day')
}

setupLifecycle().catch((e) => {
  console.error('[setup] Failed to configure R2 lifecycle rule:', e)
  process.exit(1)
})
```

**Por qué script TypeScript y no manual en el Dashboard**: el script es reproducible, versionado en git, y puede ejecutarse en cualquier entorno (dev, staging, prod) sin depender de que alguien recuerde los pasos. Configura exactamente los mismos parámetros cada vez.

**Agregar script al `package.json`** (via `pnpm pkg set`):
```json
"scripts": {
  "setup:r2-lifecycle": "tsx scripts/setup-r2-lifecycle.ts"
}
```

**Por qué `tsx`**: el script importa módulos TypeScript del proyecto (`src/lib/r2`, `src/lib/env`). `tsx` ejecuta TypeScript directamente sin compilar. Si no está instalado: `pnpm add -D tsx`.

**Verificación de la regla en R2 Dashboard** (después de ejecutar el script):
- Cloudflare Dashboard → R2 → [bucket] → Settings → Lifecycle Rules
- Debe aparecer la regla `cleanup-uploads` con prefijo `uploads/` y expiración de 1 día.

**Por qué el prefijo es `uploads/` y no el bucket completo**: el bucket puede contener otros objetos que no son uploads de usuarios (assets estáticos, configuración). El prefijo acota la regla solo a los uploads temporales.

**Criterio de aceptación**: `pnpm setup:r2-lifecycle` ejecuta sin errores. En el R2 Dashboard aparece la regla con los parámetros correctos. Ejecutar el script dos veces no crea una segunda regla — `PutBucketLifecycleConfigurationCommand` reemplaza las reglas existentes (es destructivo si hay otras reglas). Si el bucket ya tiene otras lifecycle rules, deben incluirse todas en el array `Rules` al ejecutar el script.

### T14.3 — Integrar `deleteR2Object` en `/api/analyze`

Editar `src/pages/api/analyze.ts` (creado en F13) para agregar la llamada de eliminación después del análisis exitoso.

**Posición exacta en el handler**:

```ts
// Después de runAnalysisPipeline exitoso y recordUsage:
await recordUsage(ip)
if (usage.shouldBan) await banIP(ip)

await deleteR2Object(body.objectKey)  // fail-open — no afecta la respuesta si falla

return ok(results)
```

**Por qué después de `recordUsage`**: el registro de uso debe ocurrir independientemente de si la eliminación falla. Si el orden fuera eliminación → recordUsage, un fallo de eliminación podría (en teoría) interrumpir el registro de uso si fuera una función que lanzara. Como `deleteR2Object` es fail-open, el orden es en realidad indiferente — pero documentarlo después de `recordUsage` deja claro que la eliminación es la última operación antes del return.

**Por qué no eliminar en el path de error del pipeline**: si el análisis falla, el usuario puede reintentar. Si elimináramos el objeto en el path de error, el usuario tendría que volver a subir la imagen para cada reintento. El objeto huérfano se limpia dentro de 24h por el lifecycle rule.

**Criterio de aceptación**: después de un `POST /api/analyze` exitoso, intentar `HeadObjectCommand` con el mismo `objectKey` retorna `NotFound`. El endpoint sigue retornando 200 con `AnalysisResults`.

**Verificación de la lifecycle rule**:

- ID: `cleanup-uploads`
- Status: Enabled
- Prefix: `uploads/`

Si el bucket tiene objetos fuera del prefijo `uploads/` (por ejemplo `assets/logo.png`), verificar que la regla NO los afecta. La regla solo actúa sobre `uploads/`.

**Verificación end-to-end en F19**: crear un objeto de prueba en `uploads/` con más de 24h de antigüedad (usando la API S3 con `--metadata` de timestamp en el pasado si es posible, o esperar 24h reales) y confirmar que R2 lo elimina automáticamente. ## Riesgos

| Riesgo | Mitigación |
|---|---|
| El bucket tiene otras lifecycle rules de AzanoLabs que `PutBucketLifecycleConfigurationCommand` sobreescribe | Antes de ejecutar el script, leer las reglas existentes con `GetBucketLifecycleConfigurationCommand` e incluirlas en el array `Rules` junto con la nueva. Documentar este paso en el script |
| `tsx` no está instalado como devDependency | Verificar en `package.json`. Si no está: `pnpm add -D tsx` |
| R2 no soporta `PutBucketLifecycleConfigurationCommand` (feature no disponible en todas las cuentas) | Si el comando falla con `NotImplemented`, configurar la regla manualmente en el R2 Dashboard con los mismos parámetros. Documentar en `F14-hallazgos.md` |
| La eliminación síncrona agrega latencia percibida por el usuario | ~50-150ms adicionales después de ~8s de pipeline. Impacto < 2%. Si fuera un problema, mover la eliminación antes del pipeline (eliminar el objeto anterior si el usuario sube una segunda vez) — pero cambia la lógica |
| `objectKey` en el body del request podría ser manipulado para eliminar objetos ajenos | Los objectKeys se generan con UUID en el servidor (F11) — no son predecibles. Si se quisiera blindar completamente: firmar el `objectKey` en el presign endpoint y verificar la firma en `/api/analyze` antes de eliminar |
