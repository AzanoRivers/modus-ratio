# 29062026-f8 — Estado global (Zustand) + persistencia localStorage

> Sub-plan de la **Fase 8** del fundational plan (`29062026-fundation-plan.md`).
> Migra el estado local de `HomeForm` a un store Zustand persistido en localStorage, agrega la máquina de estados de fase de la app y construye la ventana deslizante de límite de uso (25 análisis/hora).

- **Fase**: F8 - Estado global (Zustand) + persistencia localStorage
- **Fecha**: 29-06-2026
- **Estado**: cerrada
- **Prerequisitos**: F7 cerrada (HomeForm con useState funcional)
- **Bloquea**: F11 (gating de uso en subida), F17 (orquestación de flujo completo)

---

## Objetivo

Pasar de un formulario con estado efímero (se pierde al recargar) a uno con estado persistido (los parámetros se recuerdan entre sesiones). Además, implementar la máquina de estados de la app (`idle → uploading → analyzing → results → error`) y el límite de uso de 25 análisis/hora con ventana deslizante, exponiéndolo con `remaining` y `resetAt` para que la UI lo consuma en F17.

| Decisión | Valor |
|---|---|
| Librería de estado | Zustand (ya instalada en F1) |
| Persistencia | Middleware `persist` de Zustand con `partialize` (solo los campos necesarios) |

| Ventana de tiempo | 1 hora (`3_600_000` ms) — en constante `USAGE_WINDOW_MS` |
| localStorage key | `'modus-ratio-store'` |
| Esquema versionado | `version: 1` — si el esquema cambia en fases futuras se usa `migrate` |
| `currentFile` | En el store pero **no persistido** (File no es serializable) |
| `phase`, `results`, `error` | En el store pero **no persistidos** (siempre empiezan en su estado inicial al recargar) |

export const USAGE_LIMIT      = 25

**Criterio de aceptación**: `src/lib/constants.ts` exporta las 7 constantes. `src/lib/index.ts` (barrel) las re-exporta para que los consumers importen desde `@/lib`.

### T8.3 — Store Zustand con `persist`

Crear el store en `src/store/appStore.ts`:

```ts
import { create } from 'zustand'
import { persist } from 'zustand/middleware'
import {
  USAGE_LIMIT,
  USAGE_WINDOW_MS,
  STORE_VERSION,
  STORE_KEY,
} from '@/lib/constants'

// Función pura — se exporta para testearla si hace falta
export function pruneTimestamps(timestamps: number[], now: number): number[] {
  return timestamps.filter(ts => ts > now - USAGE_WINDOW_MS)
}

const initialFormParams: FormParams = {
  style:     null,
  height:    null,
  skinColor: null,
  build:     { upper: null, lower: null },
  gender:    false,
}

export const useAppStore = create<AppState>()(
  persist(
    (set, get) => ({

      // Parámetros del formulario
      formParams: initialFormParams,
      setFormParam: (key, value) =>
        set(state => ({
          formParams: { ...state.formParams, [key]: value },
        })),
      resetFormParams: () => set({ formParams: initialFormParams }),

      // Fase
      phase: 'idle',
      setPhase: (phase) => set({ phase }),

      // Archivo actual
      currentFile: null,
      setCurrentFile: (file) => set({ currentFile: file }),

      // Resultados
      results: null,
      setResults: (results) => set({ results }),

      // Error
      analysisError: null,
      setAnalysisError: (key) => set({ analysisError: key }),

      // Ventana deslizante de uso
      usageTimestamps: [],

      recordUsage: () => {
        const now = Date.now()
        set(state => ({
          usageTimestamps: [...pruneTimestamps(state.usageTimestamps, now), now],
        }))
      },

      canAnalyze: () => {
        const pruned = pruneTimestamps(get().usageTimestamps, Date.now())
        return pruned.length < USAGE_LIMIT
      },

      remaining: () => {
        const pruned = pruneTimestamps(get().usageTimestamps, Date.now())
        return Math.max(0, USAGE_LIMIT - pruned.length)
      },

      resetAt: () => {
        const pruned = pruneTimestamps(get().usageTimestamps, Date.now())
        if (pruned.length === 0) return null
        return pruned[0] + USAGE_WINDOW_MS
      },

    }),
    {
      name: STORE_KEY,
      version: STORE_VERSION,
      // Solo persistir los campos necesarios — excluye currentFile, phase, results, analysisError
      partialize: (state) => ({
        formParams:      state.formParams,
        usageTimestamps: state.usageTimestamps,
      }),
      // Si el esquema cambia en fases futuras, agregar migrate aquí
      migrate: (persisted, version) => {
        // version 0 → 1: no hay cambios aún
        return persisted as AppState
      },
    }
  )
)
```

**Por qué `pruneTimestamps` es una función pura exportada**:
- No depende del store: recibe `timestamps` y `now` como argumentos.
- Se llama dentro de `recordUsage`, `canAnalyze`, `remaining` y `resetAt` con `Date.now()` en cada llamada para que el tiempo sea siempre actual.
- Al ser pura y exportada, el Revisor puede verificarla en consola sin montar el store.
- `Date.now()` se pasa como argumento (no se llama dentro de `pruneTimestamps`) para facilitar testing: se puede pasar un timestamp fijo.

**Por qué `canAnalyze`, `remaining` y `resetAt` son funciones y no valores**: el conteo de timestamps vivos cambia con el tiempo (los viejos expiran). Si fueran valores en el estado, habría que actualizar el estado periódicamente. Al ser funciones que calculan sobre el estado en el momento de ser llamadas, siempre reflejan el conteo real sin suscripciones a timers.

**Criterio de aceptación**: el store carga en la app sin errores. `localStorage.getItem('modus-ratio-store')` en la consola muestra solo `formParams` y `usageTimestamps`, nunca `currentFile`, `phase` ni `results`.

### T8.4 — Migrar `HomeForm` de `useState` a `useAppStore`

Editar `src/components/organisms/HomeForm/HomeForm.tsx` para eliminar el `useState` de F7 y reemplazarlo con el store.

**Cambios concretos**:

1. Eliminar el `useState<FormState>` y la función `update`.
2. Importar `useAppStore` desde `@/store/appStore`.
3. Reemplazar cada acceso a `form.X` con un selector del store:

```ts
// Antes (F7):
const [form, setForm] = useState<FormState>(initialState)

// Después (F8):
const formParams    = useAppStore(state => state.formParams)
const setFormParam  = useAppStore(state => state.setFormParam)
const currentFile   = useAppStore(state => state.currentFile)
const setCurrentFile = useAppStore(state => state.setCurrentFile)
const canAnalyze    = useAppStore(state => state.canAnalyze)
```

4. Reemplazar `update('style', v)` con `setFormParam('style', v)`.
5. Reemplazar `update('file', f)` con `setCurrentFile(f)` (el archivo va a su propio campo, no a `formParams`).
6. El botón de análisis usa `canAnalyze()` además de los otros checks:

```ts
disabled={!currentFile || !formParams.style || !canAnalyze()}
```

**Por qué selectores individuales y no `useAppStore()` completo**: llamar `useAppStore()` sin selector hace que el componente se re-renderice en cualquier cambio del store (incluyendo cambios de `phase` que no afectan al formulario). Los selectores individuales acotan el re-render al campo exacto que usa el componente.

**Criterio de aceptación**: el comportamiento visual del formulario es idéntico al de F7. Recargar la página carga los parámetros guardados. El formulario NO muestra el archivo previo (porque `currentFile` no se persiste, correcto).

## Manejo de hidratación SSR → cliente

Zustand con `persist` puede causar un mismatch de hidratación en Astro SSR: el servidor renderiza el formulario con el estado inicial (vacío), pero el cliente tiene datos en localStorage. Al hidratar, React detecta la diferencia y re-renderiza.

**Comportamiento esperado y aceptado**: el formulario se renderiza vacío en el servidor (SSR), luego en < 300ms el cliente carga el localStorage y re-renderiza con los valores guardados. Visualmente es un flash rápido de estado vacío → estado guardado.

**Si el flash es perceptible y molesto**: agregar un estado `hasHydrated` al store:

```ts
// En el store:
hasHydrated: false,
setHasHydrated: (v: boolean) => set({ hasHydrated: v }),

// En persist middleware:
onRehydrateStorage: () => (state) => {
  state?.setHasHydrated(true)
}
```

Y en `HomeForm`, renderizar un `<Skeleton>` hasta que `hasHydrated === true`. Esto está documentado como mejora opcional; solo implementarlo si el Revisor detecta que el flash es notable.

## Riesgos

| Riesgo | Mitigación |
|---|---|
| `File` object en `currentFile` serializado en localStorage por error | El `partialize` excluye `currentFile` explícitamente. El Revisor verifica en Application > LocalStorage |
| El middleware `persist` de Zustand tiene una API diferente en Zustand v5 vs v4 | Confirmar la versión instalada en F1 y usar la API de esa versión. Si Zustand v5 cambió la API de `persist`, ajustar según docs |
| Timestamps manipulados por el usuario en localStorage (evasión del límite cliente) | Esperado y documentado: el localStorage es la capa UX. El espejo servidor en Redis (F10) es la protección real. No hay que blindar el localStorage; es suficiente que funcione correctamente para el caso honesto |
| `canAnalyze()`, `remaining()`, `resetAt()` calculan `Date.now()` en cada llamada y pueden retornar valores distintos en renders rápidos consecutivos | Valores consistentes dentro del mismo frame de render. La diferencia de unos milisegundos entre llamadas no afecta la UX |
| `migrate` vacío en la versión 1 del store puede producir un warning en Zustand | Agregar un comentario en el `migrate` que explique que la versión 0→1 no necesita transformación. Si Zustand lanza warning, devolver `persisted` cast como `AppState` es suficiente |
