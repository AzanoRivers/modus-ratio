## Objetivo

Producir tres páginas de error completamente funcionales y estilizadas, compartiendo un template Astro reutilizable (`ErrorDisplay.astro`). Además, formalizar `BaseLayout.astro` (que hasta ahora era un placeholder de F1) para que todas las páginas de la app compartan fuentes, variables CSS globales y la transición de entrada CRT automática.

### T9.2 — `BaseLayout.astro` — de placeholder a implementación

Reemplazar el placeholder de F1 con la implementación completa.

**Responsabilidades del layout**:
- Importar fuentes `@fontsource` (cargadas una vez, compartidas por todas las páginas).
- Importar `globals.css` y `constants.css` globales.
- Renderizar `<html lang>` con el idioma recibido como prop.
- Aplicar las clases de fondo y fuente base en `<body>`.
- Envolver el contenido en `<main class="page-enter">` para la transición CRT automática de F5.

**Props del layout**:

```ts
interface Props {
  title: string
  lang: 'es' | 'en'
  description?: string
}
```

**Estructura del layout** (pseudocódigo — el Implementador escribe el Astro real):

```
imports:
  @fontsource/orbitron/400.css
  @fontsource/orbitron/700.css
  @fontsource/audiowide/400.css
  @fontsource/space-grotesk/400.css
  @fontsource/space-grotesk/700.css
  ../styles/globals.css
  ../styles/constants.css

frontmatter:
  const { title, lang, description } = Astro.props

html:
  <html lang={lang}>
    <head>
      <meta charset="utf-8" />
      <meta name="viewport" content="width=device-width" />
      <meta name="description" content={description ?? ''} />
      <title>{title}</title>
    </head>
    <body class="font-space-grotesk bg-background">
      <main class="page-enter">
        <slot />
      </main>
    </body>
  </html>
```

**Por qué `bg-background` en `<body>` y `page-enter` en `<main>`**:
- `bg-background` garantiza que el color `--background: #041528` cubre toda la pantalla incluido el área fuera del main (scroll bounce en iOS, resizes).
- `.page-enter` aplica la animación CRT solo al contenido scrollable, no al viewport entero.

**Por qué el layout NO tiene `<nav>` ni `<footer>`**: Modus Ratio es una single-feature app. No hay navegación entre secciones. Si en el futuro se agrega, se pasa como slot nombrado.

**Nota sobre `globals.css` y los `@fontsource` importados juntos**: en Astro, importar el mismo CSS desde múltiples componentes no lo duplica en el bundle. Aun así, centralizar en el layout es la práctica estándar.

**Criterio de aceptación**: `src/pages/index.astro` usa `BaseLayout` con `title` y `lang` correctos y la app sigue compilando. La página muestra la fuente Space Grotesk y el fondo `#041528`.

### T9.5 — `src/pages/500.astro`

En Astro 5+, `src/pages/500.astro` se sirve automáticamente cuando hay un error no capturado en el servidor.

**Diferencias con 404**:
- `Astro.props` expone `{ error: unknown }` — el error que causó el 500.
- `Astro.request` puede no tener todos los headers habituales dependiendo de en qué punto del ciclo SSR se produjo el error.
- El `getLocale` puede fallar si el error ocurrió antes de que los headers estuvieran disponibles.

**Estrategia de locale resiliente**:

```astro
---
let lang: 'es' | 'en' = 'es'  // fallback por defecto
try {
  lang = getLocale(Astro.request)
} catch {
  // si getLocale falla en el contexto de error, usar fallback silencioso
}
const t = getDictionary(lang)
const e = t.errors.serverError
---
```

**Por qué fallback a `'es'` y no a `'en'`**: el proyecto está basado en España/Latinoamérica. Si la detección falla, el idioma más probable del usuario es español.

```astro
<BaseLayout title={`${e.code} · Modus Ratio`} lang={lang}>
  <ErrorDisplay
    code={e.code}
    title={e.title}
    message={e.message}
    cta={e.cta}
    colorClass="neon-text-pink"
  />
</BaseLayout>
```

**Criterio de aceptación**: el archivo existe y Astro no reporta errores de compilación. La verificación funcional de esta página se hace en F19 (QA de deploy en Vercel), ya que simular un 500 en dev requiere código que lance intencionalmente.

### T9.6 — `src/pages/429.astro`

Página servida cuando el middleware de F10 detecta una IP baneada en Redis y redirige a `/429`.

**Diferencias clave con 404/500**:
- Es una ruta normal (`/429`), no un error handler de Astro.
- El status HTTP retornado DEBE ser 429, no 200. Usar `Astro.response.status = 429` en el frontmatter.
- No muestra tiempo de espera ni información sobre el ban (es una suspensión de seguridad, no un cupo de uso).

```astro
---
import BaseLayout from '@/layouts/BaseLayout.astro'
import ErrorDisplay from '@/components/templates/ErrorDisplay/ErrorDisplay.astro'
import { getLocale, getDictionary } from '@/i18n'

Astro.response.status = 429

const lang = getLocale(Astro.request)
const t = getDictionary(lang)
const e = t.errors.tooManyRequests
---

<BaseLayout title={`${e.code} · Modus Ratio`} lang={lang}>
  <ErrorDisplay
    code={e.code}
    title={e.title}
    message={e.message}
    cta={e.cta}
    colorClass="neon-text-green"
  />
</BaseLayout>
```

**Criterio de aceptación**: navegar a `/429` directamente muestra la página. El status HTTP en Network tab es 429. El texto no menciona ningún tiempo de espera.

### T9.7 — Dependencia con F10 (documentada, no implementada aquí)

El flujo completo de la página 429 requiere que el **middleware de Astro** (implementado en F10) detecte la IP baneada en Redis y emita el redirect. F9 solo crea la página destino.

**Flujo esperado** (documentado para F10):

```
Request HTTP de cualquier ruta
  → middleware.ts (F10)
      → redis.get('modusratio:ip:<IP>:banned')
      → si existe: Response.redirect('/429', 302)
      → si no: next()
```

**Verificación de F9 sin F10**: acceder directamente a `/429` en el browser durante F9. La verificación del flujo completo (ban → redirect → página) es tarea de F10.

## Riesgos

| Riesgo | Mitigación |
|---|---|
| Astro < 5 no tiene soporte para `500.astro` | Verificar la versión de Astro instalada. Si no lo soporta, crear una ruta API que capture errores en su lugar, o dejarlo como placeholder documentado |
| `Astro.response.status = 429` puede no funcionar en `/429.astro` porque Astro puede no exponer `Astro.response` en rutas regulares | Alternativa: retornar un `Response` directamente desde el frontmatter con `return new Response(body, { status: 429 })`. El Implementador verifica cuál API funciona en la versión instalada |
| `getLocale` puede fallar en el contexto de `500.astro` | Try/catch con fallback a `'es'` (ya documentado en T9.5) |
| `colorClass` como string desnudo puede aceptar cualquier string, no solo los válidos | Tipar como `'neon-text-blue' \| 'neon-text-pink' \| 'neon-text-green'` en el componente. Si Astro no tiene prop typing estricto para `.astro`, usar un comentario JSDoc |
| El redirect 302 desde F10 a `/429` puede causar un loop si el middleware también intercepta la ruta `/429` | El middleware de F10 debe excluir `/429` del chequeo de Redis ban (o chequear solo rutas de API, no rutas de página) |
