# 29062026-f11 — R2 direct upload (presign + cliente)

> Sub-plan de la **Fase 11** del fundational plan (`29062026-fundation-plan.md`).
> Implementa el flujo completo de subida directa de imágenes a Cloudflare R2: la imagen nunca pasa por Vercel. El servidor genera una URL presignada; el cliente la usa para hacer PUT directo a R2.

- **Fase**: F11 - R2 direct upload
- **Fecha**: 29-06-2026
- **Estado**: cerrada
- **Prerequisitos**: F8 (Zustand store), F10 (helpers de infra: `env`, `getIP`, `checkUsage`, `ok/err`)
- **Bloquea**: F12 (el pipeline GPT-4o-mini necesita el `objectKey` del archivo subido), F15 (loader necesita `onProgress`)

---

## Objetivo

Cuando el usuario haga clic en "Analizar" (F17), el flujo de subida es:

```
Client → POST /api/upload/presign (metadata) → Server genera presignedUrl + objectKey
Client → PUT <presignedUrl> (bytes del archivo) → R2 directamente
Client guarda objectKey en store → F12 lo usa para pasar la URL del archivo a la IA
```

F11 entrega: el endpoint de presign, la validación del archivo, el cliente de R2, y la función cliente de subida con soporte de progreso para F15.

## Decisiones ya tomadas

| Decisión | Valor |
|---|---|
| Storage | Cloudflare R2 — API compatible con S3 |
| Biblioteca S3 | `@aws-sdk/client-s3` + `@aws-sdk/s3-request-presigner` |
| Método de subida | `PUT` presignado — el cliente sube directamente sin proxy |
| Expiración de URL presignada | 5 minutos (300 s) — suficiente para subidas lentas |
| Tamaño máximo | 10 MB (`10_485_760` bytes) |
| MIME types permitidos | `image/jpeg`, `image/png`, `image/webp`, `image/gif`, `image/heic`, `image/heif` |
| Formato de object key | `uploads/YYYYMMDD/<uuid>.<ext>` — prefijo de fecha para limpieza en F14 |
| Content-Type en presigned URL | Fijo en la URL: R2 rechaza un PUT con un Content-Type distinto al firmado |
| Progreso de subida | `XMLHttpRequest` con callback `onProgress` — `fetch` no expone progreso nativo |
| `recordUsage` / `checkUsage` | Se llaman en `/api/analyze` (F12), NO en `/api/upload/presign` — la unidad de consumo es el análisis, no la URL presignada |
| Dónde guardar `objectKey` | En el Zustand store (`currentFileKey`) — F12 lo lee para pasarlo a la IA |

## Cierre — Definition of Done

- `POST /api/upload/presign` con metadata válida retorna `{ data: { presignedUrl, objectKey } }`.
- `POST /api/upload/presign` con MIME inválido retorna `err('INVALID_MIME_TYPE', 415)`.
- `POST /api/upload/presign` con tamaño excedido retorna `err('FILE_TOO_LARGE', 413)`.
- El PUT a la `presignedUrl` retornada sube el archivo a R2 con status 200.
- Después de la subida, el archivo existe en R2 en la key `uploads/YYYYMMDD/<uuid>.<ext>`.
- `uploadToR2(file, url, onProgress)` llama `onProgress` con valores entre 0 y 100 durante la subida.
- `currentFileKey` en el store tiene el `objectKey` después de una subida exitosa.
- `pnpm build` sin errores TypeScript.

### T11.1 — Dependencias + `src/lib/r2.ts`

**Verificación de dependencias** (instaladas en F1 o instalar aquí):
```
pnpm add @aws-sdk/client-s3 @aws-sdk/s3-request-presigner
```

**`src/lib/r2.ts`** — S3Client configurado para el endpoint de R2:

```ts
import { S3Client } from '@aws-sdk/client-s3'
import { env } from '@/lib/env'

export const r2 = new S3Client({
  region: 'auto',
  endpoint: `https://${env.r2.accountId}.r2.cloudflarestorage.com`,
  credentials: {
    accessKeyId:     env.r2.accessKeyId,
    secretAccessKey: env.r2.secretAccessKey,
  },
})
```

**Por qué `region: 'auto'`**: R2 no usa regiones AWS. El SDK de S3 requiere un valor; `'auto'` es el valor que Cloudflare documenta para R2.

**Por qué no `createPresignedPost` (POST) sino `getSignedUrl` (PUT)**: el PUT presignado es más simple para subidas de un solo archivo. POST es necesario para subidas multipart (archivos muy grandes). Las fotos de outfit no superan 10 MB, así que PUT es suficiente.

**Criterio de aceptación**: el cliente `r2` se instancia sin error. El módulo compila sin errores TypeScript.

### T11.2 — `src/lib/fileValidation.ts` — Validación de archivo

```ts
export const ALLOWED_MIME_TYPES = [
  'image/jpeg',
  'image/png',
  'image/webp',
  'image/gif',
  'image/heic',
  'image/heif',
] as const

export type AllowedMimeType = (typeof ALLOWED_MIME_TYPES)[number]

export const MAX_FILE_SIZE_BYTES = 10_485_760  // 10 MB

const MIME_TO_EXT: Record<AllowedMimeType, string> = {
  'image/jpeg': 'jpg',
  'image/png':  'png',
  'image/webp': 'webp',
  'image/gif':  'gif',
  'image/heic': 'heic',
  'image/heif': 'heif',
}

export function isAllowedMime(mime: string): mime is AllowedMimeType {
  return (ALLOWED_MIME_TYPES as readonly string[]).includes(mime)
}

export function getExtension(mime: AllowedMimeType): string {
  return MIME_TO_EXT[mime]
}
```

**Por qué `HEIC` y `HEIF` como tipos separados**: iOS puede reportar cualquiera de los dos según la versión. El Dropzone de F6 acepta ambos. La validación del servidor debe aceptar ambos también para no rechazar archivos legítimos de iPhone.

**Por qué `as const` en `ALLOWED_MIME_TYPES`**: permite que TypeScript infiera `AllowedMimeType` como unión literal, evitando el cast manual.

**Criterio de aceptación**: `isAllowedMime('image/jpeg')` → `true`. `isAllowedMime('application/pdf')` → `false`. `getExtension('image/heic')` → `'heic'`.

### T11.3 — `src/lib/presign.ts` — Generador de presigned URL

```ts
import { PutObjectCommand } from '@aws-sdk/client-s3'
import { getSignedUrl } from '@aws-sdk/s3-request-presigner'
import { r2 } from '@/lib/r2'
import { env } from '@/lib/env'
import { getExtension, type AllowedMimeType } from '@/lib/fileValidation'

export interface PresignResult {
  presignedUrl: string
  objectKey:    string
}

export async function generatePresignedPut(
  contentType: AllowedMimeType,
): Promise<PresignResult> {
  const date = new Date().toISOString().slice(0, 10).replace(/-/g, '')
  const uuid = crypto.randomUUID()
  const ext  = getExtension(contentType)
  const objectKey = `uploads/${date}/${uuid}.${ext}`

  const command = new PutObjectCommand({
    Bucket:      env.r2.bucketName,
    Key:         objectKey,
    ContentType: contentType,
  })

  const presignedUrl = await getSignedUrl(r2, command, { expiresIn: 300 })

  return { presignedUrl, objectKey }
}
```

**Por qué el `objectKey` incluye la fecha (`YYYYMMDD`) y no solo el UUID**: F14 (R2 cleanup) puede listar objetos prefijados por fecha y eliminar los de días anteriores sin escanear todo el bucket. Con solo UUID no hay forma eficiente de encontrar archivos viejos.

**Por qué `ContentType` en el `PutObjectCommand`**: R2 verifica que el PUT se haga con el mismo `Content-Type` firmado. Si el cliente intenta subir con un tipo distinto, R2 rechaza el request. Esto previene que alguien consiga una URL presignada para `image/jpeg` y luego suba un ejecutable.

**Por qué no se pasa `ContentLength` al comando**: el SDK de AWS recomienda no incluir `ContentLength` en las URLs presignadas para S3/R2 PUT, ya que el cliente debe enviarlo en el header al momento del upload. Incluirlo en el comando del presign requiere que el cliente use exactamente el mismo valor, lo que puede causar errores con comprensión de red. La validación de tamaño se hace en el endpoint de presign (a partir de los datos del cliente) y el Dropzone de F6 ya filtra en cliente.

**Sobre `crypto.randomUUID()`**: disponible en Node.js 19+ y en el runtime de Vercel. Si la versión de Node es más antigua, usar `uuid` package (`pnpm add uuid`). El Implementador verifica el runtime de Vercel del proyecto.

**Criterio de aceptación**: `generatePresignedPut('image/jpeg')` retorna un objeto con `presignedUrl` (string con `X-Amz-Signature` en la URL) y `objectKey` con formato `uploads/YYYYMMDD/<uuid>.jpg`.

### T11.4 — `src/pages/api/upload/presign.ts` — Endpoint presign

**Contrato del endpoint**:
- Método: `POST`
- Body: `{ contentType: string, contentLength: number }`
- Respuesta 200: `{ data: { presignedUrl: string, objectKey: string } }`
- Respuesta 400: body inválido o campos faltantes
- Respuesta 413: archivo demasiado grande
- Respuesta 415: MIME type no permitido
- Respuesta 502: error al generar la URL presignada en R2

El endpoint recibe SOLO metadata del archivo, nunca el contenido.

```ts
import type { APIRoute } from 'astro'
import { ok, err } from '@/lib/apiResponse'
import { isAllowedMime, MAX_FILE_SIZE_BYTES } from '@/lib/fileValidation'
import { generatePresignedPut } from '@/lib/presign'
import { getIP } from '@/lib/getIP'

interface PresignRequestBody {
  contentType:   string
  contentLength: number
}

function isValidBody(body: unknown): body is PresignRequestBody {
  return (
    typeof body === 'object' &&
    body !== null &&
    typeof (body as Record<string, unknown>).contentType === 'string' &&
    typeof (body as Record<string, unknown>).contentLength === 'number'
  )
}

export const POST: APIRoute = async ({ request }) => {
  let body: unknown
  try {
    body = await request.json()
  } catch {
    return err('INVALID_JSON', 'Request body is not valid JSON', 400)
  }

  if (!isValidBody(body)) {
    return err('BAD_REQUEST', 'Missing contentType or contentLength', 400)
  }

  const { contentType, contentLength } = body

  if (!isAllowedMime(contentType)) {
    return err('INVALID_MIME_TYPE', 'File type not allowed', 415)
  }

  if (contentLength > MAX_FILE_SIZE_BYTES) {
    return err('FILE_TOO_LARGE', 'File exceeds maximum size of 10 MB', 413)
  }

  let result: { presignedUrl: string; objectKey: string }
  try {
    result = await generatePresignedPut(contentType)
  } catch {
    return err('R2_UPLOAD_FAILED', 'Could not generate upload URL', 502)
  }

  return ok(result)
}
```

**Por qué el endpoint no llama a `checkUsage` ni `recordUsage`**: el consumo de cuota se registra en `/api/analyze` (F12). Si se registrara en presign, un usuario que obtiene la URL pero no completa la subida consumiría cuota innecesariamente. Un usuario que presigna pero no analiza no consume recursos de IA.

**Por qué no se valida el `filename` del cliente**: la key se genera en el servidor con UUID. El nombre original del archivo no se usa para nada del lado del servidor, evitando cualquier path traversal o sanitización necesaria.

**Criterio de aceptación**: probar con `curl -X POST /api/upload/presign -H "Content-Type: application/json" -d '{"contentType":"image/jpeg","contentLength":1048576}'` → retorna `presignedUrl` y `objectKey`. Probar con `contentType: "application/pdf"` → retorna 415.

### T11.5 — `src/lib/client/uploadToR2.ts` — Función de subida cliente

Esta función se ejecuta en el browser. No importa nada de `src/lib/env.ts` ni otros módulos servidor. Usa `XMLHttpRequest` para exponer progreso de subida.

**Por qué XHR en lugar de `fetch`**: `fetch` no expone eventos de progreso para el upload body. `XHR` tiene `xhr.upload.onprogress` que permite actualizar una barra de progreso en tiempo real. F15 (energetic loader) consume este callback.

```ts
export interface UploadOptions {
  onProgress?: (percent: number) => void
}

export async function uploadToR2(
  file: File,
  presignedUrl: string,
  options: UploadOptions = {},
): Promise<void> {
  return new Promise((resolve, reject) => {
    const xhr = new XMLHttpRequest()

    xhr.upload.onprogress = (event) => {
      if (event.lengthComputable && options.onProgress) {
        const percent = Math.round((event.loaded / event.total) * 100)
        options.onProgress(percent)
      }
    }

    xhr.onload = () => {
      if (xhr.status >= 200 && xhr.status < 300) {
        resolve()
      } else {
        reject(new Error(`R2 upload failed with status ${xhr.status}`))
      }
    }

    xhr.onerror = () => {
      reject(new Error('R2 upload failed: network error'))
    }

    xhr.open('PUT', presignedUrl)
    xhr.setRequestHeader('Content-Type', file.type)
    xhr.send(file)
  })
}
```

**Por qué la función retorna `Promise<void>` y no el `objectKey`**: la función no sabe cuál es el objectKey — lo conoce el caller (el código de F17 que llamó primero al endpoint de presign). La función solo sube el archivo a la URL dada. El caller guarda el objectKey en el store.

**Por qué `reject` con un `Error` y no un `Response`**: esta es código cliente, no un APIRoute. En cliente, rechazar con `Error` es el patrón correcto. El caller en F17 captura el error y actualiza el estado de la app.

**Criterio de aceptación**: llamar `uploadToR2(file, presignedUrl, { onProgress: console.log })` sube el archivo a R2 y loguea porcentajes de 0 a 100 en la consola. El bucket R2 muestra el archivo en la key correcta.

## Riesgos

| Riesgo | Mitigación |
|---|---|
| `crypto.randomUUID()` no disponible en el runtime de Vercel usado | Verificar versión de Node en Vercel. Si no está disponible, instalar `pnpm add uuid` y usar `v4()` de ese paquete |
| R2 rechaza el PUT porque `Content-Type` del header del cliente no coincide con el firmado | El cliente debe usar exactamente `file.type` en el header. Documentado en T11.5. Si iOS reporta un tipo distinto al que el Dropzone detectó, puede fallar — en ese caso normalizar el MIME type antes de enviarlo al presign |
| CORS no configurado antes de probar el upload: el browser bloquea el PUT | Siempre configurar CORS (T11.8) antes de la verificación de T11.9 |
| `getSignedUrl` lanza si las credenciales de R2 son inválidas | El try/catch del endpoint de presign captura este error y retorna 502. No expone la razón real al cliente |
| La presigned URL expira en 5 minutos: si el usuario tarda más en el upload (conexión lenta), falla con 403 | Para fotos de hasta 10 MB, 5 minutos es más que suficiente incluso en 3G (2 Mbps ≈ 40s). No escalar el TTL sin una razón real |
| El mismo cliente hace spam de llamadas a `/api/upload/presign` para generar muchas URLs y "reservar" cuota | El middleware de F10 ya aplica el ban por rate de requests. Sin embargo, el presign no consume cuota, así que el spam de presign solo genera keys huérfanas en R2, que F14 limpiará |
