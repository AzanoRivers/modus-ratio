## Correcciones post-cierre (01-07-2026, primera prueba real end-to-end)

F13 se cerró sin haber probado nunca la llamada real a Minimax M3 (ver [[feedback-minimax-plans]]). La primera prueba real, después de F17/F18/F17b, encontró 2 bugs genuinos en `src/lib/minimax.ts` y `src/lib/analyzeWithMinimax.ts`:

1. **`baseURL`/`MINIMAX_MODEL` incorrectos**: T13.4 usaba `https://api.opencode.ai/v1` + `'minimax/minimax-m3'` (una de las dos plantillas sin confirmar que el propio T13.4 dejaba como opciones). El valor **ya confirmado** en `features/F0-hallazgos.md` (sección D8) era `https://opencode.ai/zen/go/v1` + `'minimax-m3'`. Corregido para usar el valor de F0, que es el que sí se verificó contra la API real.
2. **Minimax M3 es un modelo de razonamiento**: emite un bloque `<think>...</think>` con su cadena de pensamiento antes del JSON final, no contemplado en el plan original. Con `max_tokens: 500` el bloque de pensamiento agotaba el límite antes de llegar a producir el JSON, causando `INVALID_RESPONSE` en el 100% de los intentos. Corregido: `extractJson()` ahora descarta el bloque `<think>` antes de parsear, y `max_tokens` subió a 3000.

**Riesgo abierto real (no solo teórico)**: con 500 tokens (solo el `<think>` truncado) la respuesta tardó ~17s. El presupuesto de 3000 tokens puede llevar la latencia real de Minimax a 30-60+ segundos según cuánto "piense" el modelo por outfit, mucho más que la estimación original de F13 (~5s). Esto agrava el riesgo de timeout de Vercel ya anotado en este mismo documento (sección Riesgos) y debe medirse con casos reales antes de F19, no asumirse resuelto por tener el plan Pro (60s).

## Rediseño de contrato (01-07-2026): texto personalizado por dimensión/warning/highlight/recomendación

Primera prueba real completa (score 69, "look con potencial") funcionó técnicamente, pero el usuario reportó que el contenido (warnings, highlight, recomendaciones) se sentía genérico y plano: eran siempre el mismo texto fijo de `es.ts`/`en.ts` por key, sin relación real con la imagen. El propio log del `<think>` de esa prueba mostraba que Minimax SÍ razona con detalle específico sobre la imagen (colores, prendas, estilo exacto), pero el esquema anterior descartaba ese razonamiento y solo se quedaba con una key de lista cerrada.

**Cambio de contrato**: `AnalysisResults.dimensions` pasa de `Record<DimensionKey, number>` a `Record<DimensionKey, {score, detail}>`. `warnings`/`recommendations` pasan de `Key[]` a `{key, detail}[]`. `highlight` pasa de `Key | null` a `{key, detail} | null`. El `detail` es una frase corta escrita por Minimax, específica a la imagen (debe referenciar una prenda/color real de la `OutfitDescription`), en el idioma del usuario (`Astro.locals.language`, propagado por primera vez hasta `analyzeWithMinimax`). Las keys cerradas se mantienen (para futura categorización/iconografía), pero el texto fijo de `t.analysis.warnings/highlights` y `t.recommendations` ya no se muestra en pantalla (queda como diccionario sin uso activo, no se borró).

**Tono del prompt**: se suavizó de "evaluador técnico, no un crítico, puntuaciones son mediciones no veredictos" a un tono de "amigo con conocimiento de moda", manteniendo el rigor SOLO en el cálculo del score, no en cómo se redacta el texto.

**Costo de esto**: `max_tokens` subió de 500 (ya insuficiente por el `<think>`) a 4000, porque ahora la respuesta incluye hasta 14 frases (5 dimensiones + 3 warnings + 1 highlight + 5 recomendaciones), no solo números y keys. Esto empeora aún más el riesgo de latencia ya anotado arriba: pendiente medir con casos reales antes de F19.

**No probado todavía contra la API real** con este nuevo esquema (cambio implementado pero sin una corrida end-to-end que lo confirme al momento de escribir esto).

## Objetivo

Minimax M3 es el motor analítico del sistema. Recibe un `OutfitDescription` detallado (de GPT-4o-mini) junto con los parámetros del usuario y el corpus de principios de moda, y produce el análisis completo: 5 dimensiones con scores, un global ratio, el semáforo, las warnings, el highlight, y las recomendaciones accionables. El endpoint `/api/analyze` (definido en esta fase) orquesta ambos modelos y retorna el `AnalysisResults` final al cliente.

### T13.1 — `src/lib/analysisTypes.ts` — Tipos y funciones de cálculo

Crear el archivo con los tipos `DimensionKey`, `WarningKey`, `HighlightKey`, `RecommendationKey` y `AnalysisResults` tal como están definidos en el apartado "Tipos finales" de este documento.

Agregar también `calcSemaphore` — única función de cálculo que vive aquí (el `globalRatio` lo produce Minimax):

```ts
import { RATIO_GREEN_MIN, RATIO_YELLOW_MIN } from '@/lib/constants'

export function calcSemaphore(ratio: number): AnalysisResults['semaphore'] {
  if (ratio >= RATIO_GREEN_MIN)  return 'green'
  if (ratio >= RATIO_YELLOW_MIN) return 'yellow'
  return 'red'
}
```

**Por qué `globalRatio` lo calcula Minimax y no el servidor**: Minimax conoce el peso relativo de cada dimensión en el contexto del outfit específico y los parámetros del usuario. Un outfit con `fitAndSilhouette` bajo puede ser más grave que uno con `originality` bajo — el modelo puede ponderar. El servidor solo se reserva el `semaphore` porque es un umbral de negocio (verde/amarillo/rojo) que no debe variar según el modelo.

**Criterio de aceptación**: `calcSemaphore(70)` → `'green'`. `calcSemaphore(69)` → `'yellow'`. `calcSemaphore(39)` → `'red'`.

### T13.5 — `src/lib/prompts/minimaxM3.ts` — Prompt de análisis completo

**System prompt — estructura obligatoria**:

1. **Rol**: experto analista de moda y estilista con conocimiento profundo de principios de styling. Recibe una descripción técnica de un outfit y datos del usuario, y produce un análisis completo y estructurado.

2. **Corpus en el prompt**: lista numerada de los 20 principios con su key y descripción breve, construida dinámicamente desde `CORPUS_DESCRIPTIONS`:
   ```
   STYLE PRINCIPLES CORPUS:
   1. color_complementary: Use colors opposite on the color wheel...
   2. color_analogous: ...
   ...
   20. hem_length_coordination: ...
   ```

3. **Instrucciones de scoring** para cada dimensión:
   - `colorHarmony`: evaluar la armonía entre los colores del outfit y la compatibilidad con el tono de piel indicado.
   - `styleCoherence`: coherencia entre todas las prendas y el estilo objetivo declarado.
   - `fitAndSilhouette`: cómo el corte favorece la silueta descrita, considerando la contextura.
   - `originality`: creatividad y personalidad del outfit dentro de su estilo.
   - `contextFit`: adecuación del look al contexto o situación del estilo objetivo.
   - Calibración: 90-100 excepcional, 70-89 bueno, 50-69 aceptable, 30-49 mejorable, 0-29 problemático. Valores enteros.

4. **`globalRatio`**: promedio ponderado de las 5 dimensiones. El modelo tiene libertad de ponderar según el outfit — no es un promedio simple si hay dimensiones más críticas.

5. **Listas cerradas** — el modelo DEBE usar exactamente estas strings, ninguna otra:
   - Warnings (hasta 3): `colors_clash`, `pattern_overload`, `ill_fitted`, `occasion_mismatch`, `accessories_missing`, `layering_issue`, `proportion_imbalance`
   - Highlight (1 o null): `great_color_combo`, `statement_piece`, `well_fitted`, `cohesive_look`, `unique_style`
   - Recommendations (hasta 5, ordenadas por prioridad): las 20 keys del corpus

6. **Schema JSON exacto de respuesta**:
   ```json
   {
     "dimensions": {
       "colorHarmony": <int 0-100>,
       "styleCoherence": <int 0-100>,
       "fitAndSilhouette": <int 0-100>,
       "originality": <int 0-100>,
       "contextFit": <int 0-100>
     },
     "globalRatio": <int 0-100>,
     "warnings": [<0-3 keys>],
     "highlight": <key o null>,
     "recommendations": [<0-5 keys del corpus, por prioridad>]
   }
   ```
   Nota: `semaphore` NO va en el JSON — lo calcula el servidor.

**Función de user prompt** — construye el mensaje con la `OutfitDescription` y los parámetros del usuario:

```ts
import type { OutfitDescription } from '@/lib/outfitDescription'
import type { FormParams } from '@/components/organisms/HomeForm'

export function buildMinimaxUserPrompt(
  description: OutfitDescription,
  formParams: FormParams,
): string
```

El user prompt incluye:
- El `OutfitDescription` serializado como JSON legible (con etiquetas claras)
- El estilo objetivo del usuario (label en inglés)
- La contextura (upper/lower en inglés)
- El tono de piel (en inglés)
- El género preferido si aplica
- La instrucción de análisis

**Por qué el `OutfitDescription` se pasa completo y no solo algunos campos**: Minimax necesita toda la información disponible para hacer un análisis coherente. Omitir campos podría hacer que ignore interacciones entre prendas que GPT-4o-mini sí detectó.

**Por qué no se pasa la imagen a Minimax**: si Minimax ya recibe el `OutfitDescription` detallado de GPT, no necesita ver la imagen de nuevo. Esto elimina la dependencia de que Minimax M3 sea multimodal (si no lo es, el pipeline sigue funcionando). Si Minimax sí es multimodal y pasa la imagen también, puede mejorar la calidad — documentar como mejora opcional si se verifica en T13.4.

**Criterio de aceptación**: el system prompt generado incluye las 20 keys del corpus en formato de lista numerada. `buildMinimaxUserPrompt(description, formParams)` retorna un string que incluye el JSON de `OutfitDescription` y los parámetros del usuario en inglés.

### T13.6 — `src/lib/analyzeWithMinimax.ts` — Función de análisis completo

```ts
import type { AnalysisResults } from '@/lib/analysisTypes'
import {
  DIMENSION_KEYS, WARNING_KEYS, HIGHLIGHT_KEYS, RECOMMENDATION_KEYS,
  calcSemaphore,
} from '@/lib/analysisTypes'
import { minimax, MINIMAX_MODEL } from '@/lib/minimax'
import { buildMinimaxSystemPrompt, buildMinimaxUserPrompt } from '@/lib/prompts/minimaxM3'
import type { OutfitDescription } from '@/lib/outfitDescription'
import type { FormParams } from '@/components/organisms/HomeForm'
import { AnalysisError } from '@/lib/analysisPipeline'

export async function analyzeWithMinimax(
  description: OutfitDescription,
  formParams:  FormParams,
): Promise<AnalysisResults>
```

**Flujo interno**:
1. Llamar a `minimax.chat.completions.create` con:
   - `model: MINIMAX_MODEL`
   - `temperature: 0.3`
   - `max_tokens: 500`
   - `response_format: { type: 'json_object' }`
   - System message: `buildMinimaxSystemPrompt()` (incluye el corpus)
   - User message: `buildMinimaxUserPrompt(description, formParams)`
2. Parsear el JSON de la respuesta
3. Validar y sanitizar:
   - Verificar que las 5 dimensiones están presentes y son enteros 0-100
   - Verificar que `globalRatio` es entero 0-100
   - Filtrar `warnings`: solo los que están en `WARNING_KEYS`
   - Filtrar `highlight`: `null` si no está en `HIGHLIGHT_KEYS`
   - Filtrar `recommendations`: solo los que están en `RECOMMENDATION_KEYS`, máximo 5
4. Calcular `semaphore` server-side con `calcSemaphore(globalRatio)`
5. Retornar `AnalysisResults` completo y tipado

**Manejo de errores — lanza `AnalysisError`**:
- Si la llamada a la API falla: `throw new AnalysisError('MINIMAX_FAILED')`
- Si el JSON no parsea: `throw new AnalysisError('INVALID_RESPONSE')`
- Si las dimensiones o `globalRatio` son inválidos: `throw new AnalysisError('INVALID_RESPONSE')`

A diferencia de la concepción anterior, `analyzeWithMinimax` lanza porque es el componente principal del análisis — sin él no hay resultado útil.

**Por qué `warnings`, `highlight` y `recommendations` se filtran en lugar de lanzar si son inválidos**: los scores son críticos (sin ellos no hay análisis), pero keys inválidas en listas son tolerables — se filtran silenciosamente y se logea un warning. El resultado sigue siendo válido con listas más cortas.

**Criterio de aceptación**: con Minimax disponible, retorna un `AnalysisResults` completo donde `semaphore` viene de `calcSemaphore`, no del modelo. Con API key inválida, lanza `AnalysisError('MINIMAX_FAILED')`.

### T13.7 — `src/lib/analysisPipeline.ts` — Pipeline completo

Orquesta la extracción de F12 y el análisis de F13 en secuencia:

```ts
import type { AnalysisResults } from '@/lib/analysisTypes'
import { extractOutfitDescription, ExtractionError } from '@/lib/extractOutfitDescription'
import { analyzeWithMinimax } from '@/lib/analyzeWithMinimax'
import type { FormParams } from '@/components/organisms/HomeForm'

export type AnalysisErrorCode =
  | 'EXTRACTION_FAILED'
  | 'MINIMAX_FAILED'
  | 'INVALID_RESPONSE'
  | 'EMPTY_RESPONSE'

export class AnalysisError extends Error {
  constructor(public readonly code: AnalysisErrorCode, message?: string) {
    super(message ?? code)
  }
}

export interface PipelineInput {
  objectKey:  string
  formParams: FormParams
}

export async function runAnalysisPipeline(
  input: PipelineInput,
): Promise<AnalysisResults>
```

**Flujo**:
1. Llamar a `extractOutfitDescription(input)` — si lanza `ExtractionError`, convertir a `AnalysisError('EXTRACTION_FAILED')` y re-lanzar.
2. Llamar a `analyzeWithMinimax(description, input.formParams)` — si lanza, re-lanzar como está (ya es `AnalysisError`).
3. Retornar el `AnalysisResults`.

**Por qué la conversión de `ExtractionError` a `AnalysisError`**: el endpoint solo necesita conocer `AnalysisError`. No importa si el fallo fue en GPT o en Minimax — el error de cara al cliente es el mismo (`AI_ANALYSIS_FAILED`). El log de servidor ya tiene el detalle.

**Criterio de aceptación**: el pipeline ejecuta ambos modelos en secuencia. Si `extractOutfitDescription` falla, `analyzeWithMinimax` no se llama (corto-circuito).

### T13.8 — `src/pages/api/analyze.ts` — Endpoint de análisis

**Contrato**:
- Método: `POST`
- Body: `{ objectKey: string, formParams: FormParams }`
- Respuesta 200: `{ data: AnalysisResults }`
- Respuesta 400: body inválido
- Respuesta 429: límite de uso alcanzado
- Respuesta 502: fallo en el pipeline de IA

```ts
import type { APIRoute } from 'astro'
import { ok, err, getIP, checkUsage, recordUsage, banIP } from '@/lib'
import { runAnalysisPipeline, AnalysisError } from '@/lib/analysisPipeline'
import type { FormParams } from '@/components/organisms/HomeForm'

export const POST: APIRoute = async ({ request, locals }) => {
  let body: unknown
  try {
    body = await request.json()
  } catch {
    return err('INVALID_JSON', 'Request body is not valid JSON', 400)
  }

  if (!isValidBody(body)) {
    return err('BAD_REQUEST', 'Missing objectKey or formParams', 400)
  }

  const ip = getIP(request)
  const usage = await checkUsage(ip)

  if (!usage.allowed) {
    if (usage.shouldBan) await banIP(ip)
    return err('USAGE_LIMIT_EXCEEDED', 'Hourly limit reached', 429)
  }

  let results: AnalysisResults
  try {
    results = await runAnalysisPipeline({
      objectKey:  body.objectKey,
      formParams: body.formParams,
    })
  } catch (e) {
    if (e instanceof AnalysisError) {
      return err('AI_ANALYSIS_FAILED', e.code, 502)
    }
    return err('AI_ANALYSIS_FAILED', 'Unexpected error', 502)
  }

  await recordUsage(ip)
  if (usage.shouldBan) await banIP(ip)

  return ok(results)
}
```

**Criterio de aceptación**: request válido → 200 con `AnalysisResults`. IP con 25 registros → 429. Pipeline fallido → 502.
