## Convenciones que aplican a TODOS los componentes de F6

Estas reglas son invariantes. El Revisor las comprueba en cada componente antes de aprobarlo.

| Regla | Detalle |
|---|---|
| Estructura de carpeta | `ComponentName/ComponentName.tsx` + `ComponentName.css` + `index.ts` |
| Sin estilos inline | `style={{}}` solo si es una CSS custom property calculada en runtime. Nunca colores, tamaños ni displays literales |
| Sin texto hardcodeado | Todo string visible viene de `t: Translations` recibido como prop |
| Variantes con clases modificadoras | La variante de un componente se aplica con clase CSS (`.badge--warning`, `.button--secondary`), nunca con objeto `style` condicional |
| Estados con clases CSS | Hover y focus en CSS (`:hover`, `:focus-visible`). Un estado activo JS usa `.componente--active` vía `className` |
| Accesibilidad mínima | `aria-label` en botones sin texto visible, `role` correcto, foco visible con outline neón |
| `prefers-reduced-motion` | Cualquier animación CSS en el componente tiene su override en el media query correspondiente |
| Props de i18n | Los componentes que muestran texto reciben `t: Translations`; los que no (Card, Skeleton) no la reciben |
| TypeScript estricto | Todas las props tipadas explícitamente; sin `any`; enums de variantes como union types |

## Tokens semánticos adicionales (T6.1 — extensión de F4)

F4 definió los colores de marca (neón azul, rosa, verde). F6 necesita colores semánticos para las variantes de `Badge` y el semáforo de resultados. Estos se agregan a `:root` en `globals.css` como extensión de F4.

```css
:root {
  /* Semánticos — añadir en la sección de colores */
  --color-success:      rgba(4,   233, 207, 0.85);  /* = neon-green ligeramente más opaco */
  --color-warning:      rgba(255, 200,  50, 0.85);  /* ámbar — nuevo, no existe en azanolabs-web */
  --color-error:        rgba(255,  60,  60, 0.85);  /* rojo — nuevo */
  --color-info:         rgba(11,  210, 255, 0.85);  /* = neon-blue ligeramente más opaco */

  /* Glow semántico para bordes y sombras */
  --color-success-glow: rgba(4,   233, 207, 0.5);
  --color-warning-glow: rgba(255, 200,  50, 0.5);
  --color-error-glow:   rgba(255,  60,  60, 0.5);
  --color-info-glow:    rgba(11,  210, 255, 0.5);
}
```

Y en el bloque `@theme` (para acceder vía Tailwind si fuera necesario):
```css
@theme {
  --color-success:      var(--color-success);
  --color-warning:      var(--color-warning);
  --color-error:        var(--color-error);
  --color-info:         var(--color-info);
}
```

Estos tokens son la fuente de verdad del semáforo: cuando en F16 se pintan las barras, se usan estas variables. Los umbrales numéricos (70 / 40) ya están en `constants.css` (F4, T4.7).

### T6.2 — Átomo: `Badge`

Badges largos para avisos, notas y alertas de privacidad. Usados en el Home (privacidad), en resultados (calidad de imagen, estilo personal) y en mensajes de error inline.

**Props**:
```ts
interface BadgeProps {
  variant: 'info' | 'warning' | 'error' | 'success'
  children: React.ReactNode   // texto ya traducido por el caller
  className?: string
}
```

**Estructura HTML esperada**:
```html
<div class="badge badge--{variant}" role="note">
  <span class="badge__icon"><!-- icono por variante --></span>
  <span class="badge__content">{children}</span>
</div>
```

**CSS — clases**:
- `.badge`: layout flex, padding, borde redondeado, font-mono, ancho completo
- `.badge--info`: borde y color `var(--color-info)`, glow `var(--color-info-glow)`
- `.badge--warning`: borde y color `var(--color-warning)`, glow `var(--color-warning-glow)`
- `.badge--error`: borde y color `var(--color-error)`, glow `var(--color-error-glow)`
- `.badge--success`: borde y color `var(--color-success)`, glow `var(--color-success-glow)`

**Iconos por variante** (caracteres Unicode, sin librería de iconos para no agregar dependencias):
- `info` → `ℹ`
- `warning` → `⚠`
- `error` → `⛔`
- `success` → `✓`

**i18n**: el `Badge` no consume `t` directamente; recibe `children` ya traducido. El caller pasa `{t.warnings.privacy}`.

**Accesibilidad**: `role="note"` en el contenedor. Si el badge es un error crítico, el caller puede agregar `role="alert"` pasándolo como prop (agregar `role?: string` a las props).

**Criterio de aceptación**: las 4 variantes renderizan con el color correcto. El icono es visible. `pnpm build` sin errores.

### T6.7 — Molécula: `Dropzone`

Área de subida de imagen con drag & drop, vista previa de la imagen seleccionada y mensajes de error de validación de formato/tamaño.

**Props**:
```ts
interface DropzoneProps {
  t: Translations
  onFileSelected: (file: File) => void
  disabled?: boolean
  className?: string
}
```

Las restricciones de formato y tamaño son constantes internas del componente (no props), ya que son fijas en el proyecto: `['image/jpeg', 'image/png', 'image/webp', 'image/heic', 'image/heif']` y `10` MB. Si cambian, se cambia el componente.

**Estados internos (con `useState`)**:
- `dragActive: boolean` — drag sobre la zona
- `preview: string | null` — URL object de la imagen seleccionada
- `error: 'format' | 'size' | null` — tipo de error de validación

**Comportamiento**:
1. El usuario arrastra un archivo o hace click para abrir el selector.
2. Al seleccionar: validar MIME type y tamaño. Si falla → mostrar `Badge` con `t.errors.imageFormat` o `t.errors.imageSize`.
3. Si pasa: crear `URL.createObjectURL(file)` para `preview`, llamar `onFileSelected(file)`.
4. La vista previa muestra la imagen en un contenedor cuadrado con `object-fit: cover`.
5. `dragActive` agrega la clase `.dropzone--drag-active` (cambia el borde a neón verde).

**Sobre HEIC**: el tipo MIME de HEIC en Safari iOS puede ser `image/heic` o `image/heif`. El componente acepta ambos. No se intenta renderizar el preview de HEIC en el navegador (WebKit lo maneja nativamente en iOS; en desktop puede no tener decoder). Si `URL.createObjectURL` genera una imagen no renderizable, el preview muestra un ícono de imagen genérico en lugar de fallar.

**CSS — clases**:
- `.dropzone`: borde dashed `var(--neon-blue)` con opacidad baja, padding, cursor pointer, transition-base
- `.dropzone--drag-active`: borde `var(--neon-green)`, glow verde, escala ligera
- `.dropzone--error`: borde `var(--color-error)`
- `.dropzone__preview`: contenedor cuadrado con la imagen
- `.dropzone__hint`: texto de instrucciones (font-mono, tamaño pequeño)

**i18n keys usadas**: `t.form.dropzoneLabel`, `t.form.dropzoneHint`, `t.form.dropzoneDrag`, `t.errors.imageFormat`, `t.errors.imageSize`

**Criterio de aceptación**: arrastrar un PNG válido muestra el preview. Arrastrar un PDF muestra el badge de error de formato. Un archivo de 15MB muestra el badge de error de tamaño. El estado `disabled` bloquea la interacción y la zona tiene opacidad reducida.

### T6.9 — Molécula: `HeightSlider`

Slider de altura de 130 a 210 cm, con display del valor actual. Optimizado para desktop y móvil (zona de touch ampliada).

**Props**:
```ts
interface HeightSliderProps {
  t: Translations
  value: number | null   // cm; null = no seleccionado
  onChange: (cm: number | null) => void
  className?: string
}
```

**Renderizado**:
```html
<div class="height-slider">
  <label class="height-slider__label">
    {t.form.heightLabel}
    <span class="height-slider__optional">({t.common.optional})</span>
  </label>
  <div class="height-slider__track-wrapper">
    <input type="range" min="130" max="210" step="1" value={value ?? 170} />
    <span class="height-slider__value">{value ? `${value} ${t.form.heightUnit}` : '—'}</span>
  </div>
</div>
```

**Por qué `value ?? 170`**: el slider HTML siempre tiene un valor numérico. Si el campo es opcional y el usuario no lo ha tocado (`value = null`), el slider muestra la posición media (170cm) pero el display del valor muestra "—". Solo cuando el usuario mueve el slider se llama `onChange(newValue)`.

**CSS del slider** (nativo, estilizado con CSS puro):
```css
input[type="range"] { /* resetear apariencia nativa */ }
input[type="range"]::-webkit-slider-thumb { /* thumb neón azul */ }
input[type="range"]::-moz-range-thumb { /* Firefox */ }
input[type="range"]::-webkit-slider-runnable-track { /* track oscuro */ }
```

**Touch target en móvil**: el thumb debe tener al menos 44×44px para ser cómodamente tocable en iPhone. Se logra con `width: 44px; height: 44px` en el pseudo-elemento del thumb, sin afectar el layout del track.

**i18n keys**: `t.form.heightLabel`, `t.form.heightUnit`, `t.common.optional`

**Criterio de aceptación**: mover el slider actualiza el display del valor en tiempo real. En móvil (DevTools), el thumb es fácilmente tocable (≥44px). `value={null}` muestra "—" y el slider en posición 170cm.

- Sección "Átomos": Badge (4 variantes), Button (primary/secondary/loading/disabled), Skeleton (4 variantes), Card (4 variantes), GenderCheckbox
- Sección "Moléculas": Dropzone (estado inicial, con preview, con error de formato, con error de tamaño), StyleSelector, HeightSlider, SkinColorSelector, BuildSelector

Cada sección es un componente React hidratado con `client:load` para que las interacciones funcionen en la demo. El catálogo usa el diccionario español por defecto (se puede cambiar pasando el locale al componente).

**Criterio de aceptación** (catálogo):
- Todos los componentes renderizan sin errores de consola.
- Ningún componente tiene estilos inline (verificable en DevTools > Elements).
- Ningún componente muestra texto hardcodeado (todo viene del diccionario).
- El catálogo pasa el DoD visual: glow correcto, hover en CSS, focus visible con outline neón.

Todas estas claves ya existen en el diccionario de F3. Si al implementar se detecta una clave faltante, el proceso es: (1) agregar en `types.ts`, (2) agregar en `es.ts`, (3) agregar en `en.ts`, (4) continuar. Nunca hardcodear el string.

## Riesgos

| Riesgo | Mitigación |
|---|---|
| El input `type="range"` en iOS Safari no permite styling del thumb con `appearance: none` sin `-webkit-appearance` | Agregar `-webkit-appearance: none` en el CSS del slider; documentar en el contexto del Implementador para que lo aplique por defecto en todos los inputs |
| HEIC preview no renderiza en Chrome/Windows (sin decoder nativo) | El Dropzone muestra un icono genérico si `<img src={preview}>` emite error `onError`. Se maneja con el evento `onError` de la imagen: clase `.dropzone__preview--no-decode` que oculta el `<img>` y muestra el ícono |
| `URL.createObjectURL` no se libera → memory leak | Llamar `URL.revokeObjectURL(preview)` en el cleanup del `useEffect` que crea el object URL |
| Los 8 chips de `StyleSelector` no caben en 1 fila en pantalla de 270px (sm) | El grid usa `grid-template-columns: repeat(2, 1fr)` en sm y `repeat(4, 1fr)` en xl, verificado en el catálogo |
| `BuildSelector` con `BuildType` igual al de `StyleSelectorOption` puede confundirse | Son tipos distintos en distintos archivos. El Implementador nunca re-exporta uno por el otro |

## Nota de ejecución

El Implementador crea los componentes en este orden estricto:
1. T6.1 (tokens semánticos) — necesario para Badge.
2. Átomos (T6.2–T6.6) — sin dependencias entre sí, pueden ejecutarse en paralelo si el Harness lo permite.
3. Moléculas (T6.7–T6.11) — requieren que los átomos existan (Dropzone usa Badge para errores; StyleSelector usa el patrón de chip de Button).
4. T6.12 (barrels + catálogo) — al final, cuando todos los componentes existen.

Antes de cerrar F6, el Revisor ejecuta el catálogo completo en los 4 breakpoints (DevTools responsive: 270px, 360px, 440px, 600px) y documenta cualquier desbordamiento visual.
