#### T0.8 — Cloudflare R2: presigned PUT + CORS (subida directa desde navegador)

**Objetivo**: confirmar el flujo exacto para subir una imagen directamente desde el navegador a R2 sin que los bytes pasen por Vercel.

**Qué investigar**:
- **SDK a usar**: R2 es compatible con el SDK S3 de AWS (`@aws-sdk/client-s3` + `@aws-sdk/s3-request-presigner`). Confirmar que estos paquetes funcionan en el servidor de Astro (Node.js / Vercel serverless) para generar la URL prefirmada.
- **Flujo presigned PUT**: el servidor Astro genera una URL prefirmada (presigned PUT URL) con un tiempo de expiración corto. El cliente hace un `PUT` directo a esa URL. Confirmar los parámetros de la función `getSignedUrl` + `PutObjectCommand` necesarios para R2.
- **CORS del bucket**: R2 permite configurar reglas CORS desde el dashboard de Cloudflare. Confirmar la regla CORS mínima necesaria para permitir `PUT` desde el dominio de Vercel (`modus.azanolabs.com`).
- **URL pública**: tras el PUT, la URL pública del objeto en R2. Confirmar el formato: `https://<bucket>.<account-id>.r2.cloudflarestorage.com/<key>` o si se usa un dominio custom con R2.
- **Credenciales de R2 para el SDK S3**: R2 genera tokens de API propios (`Access Key ID` / `Secret Access Key` + `endpoint`). Confirmar dónde se obtienen en el dashboard de Cloudflare.
- **Validaciones servidor al generar presign**: el endpoint de Astro que genera el presign debe validar formato (jpg/png/webp/heic) y tamaño (máx 10MB) antes de firmar. Confirmar cómo validar el tipo MIME y tamaño en la request sin recibir los bytes del archivo.

**Fuentes a consultar**:
- Docs de Cloudflare R2: "S3 API compatibility" y "Presigned URLs".
- Docs de `@aws-sdk/client-s3` y `@aws-sdk/s3-request-presigner`.
- Docs de Cloudflare R2: sección "CORS".

**Criterio de aceptación**:
- Flujo documentado paso a paso: (1) cliente solicita presign al endpoint Astro, (2) servidor valida y firma, (3) cliente hace PUT directo a R2, (4) URL pública disponible.
- Variables de entorno necesarias para R2 identificadas (`R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET_NAME`, `R2_ENDPOINT`).
- Regla CORS mínima del bucket documentada.
- Confirmado que el endpoint de presign puede rechazar tipos/tamaños inválidos sin recibir los bytes.

**Riesgo**: HEIC no tiene tipo MIME estándar universal (`image/heic` vs `image/heif`). Investigar cómo el navegador (especialmente Safari en iPhone) reporta el MIME type de un archivo HEIC en el `FormData` o en el header de la request, y qué validar en el servidor.
