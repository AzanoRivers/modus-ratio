## Estructura final de `.agents/`

```
.agents/
├── AGENTS.md                   ← doc maestro del Harness
├── orchestrator/
│   ├── CLAUDE.md               ← config del rol (modelo + instrucciones)
│   └── context.md              ← contexto técnico del proyecto para el Orquestador
├── implementer/
│   ├── CLAUDE.md               ← config del rol
│   └── context.md              ← contexto técnico para el Implementador
├── reviewer/
│   ├── CLAUDE.md               ← config del rol
│   └── context.md              ← contexto técnico para el Revisor
└── skills/
    └── README.md               ← inventario de skills instaladas
```

**Por qué `context.md` por rol y no uno solo**: cada rol consume información diferente del proyecto. El Orquestador necesita el mapa de fases y decisiones de arquitectura. El Implementador necesita convenciones de código, estructura de carpetas y criterios de calidad. El Revisor necesita los criterios de aceptación y los anti-patrones documentados. Un `context.md` compartido sería ruido innecesario para cada rol.

**OpenCode**: lee el mismo formato de `CLAUDE.md`. La compatibilidad es directa: los mismos archivos de rol funcionan en ambos entornos. El único cambio al hacer swap es la referencia de modelo dentro de cada `CLAUDE.md`.

### T2.1 — Crear la estructura de carpetas de `.agents/`

Crear el árbol de directorios de la sección anterior. No hay código de app aquí: solo archivos de configuración y documentación de agentes.

### T2.2 — Escribir `AGENTS.md` (doc maestro del Harness)

Este archivo es la guía de entrada para cualquier desarrollador (o agente) que necesite ejecutar una tarea del proyecto. Debe cubrir las siguientes secciones:

**Secciones que debe tener `AGENTS.md`**:

1. **Qué es el Harness**: descripción en 2–3 párrafos. El Harness es el ciclo estructurado de 4 pasos (Analizar, Orquestar, Implementar, Revisar/Auditar) que garantiza que ninguna tarea se implementa sin haber sido entendida y planificada, y sin haber sido revisada tras la implementación.

2. **Los 3 roles**:
   - **Orquestador**: analiza la tarea, identifica las tecnologías y skills necesarias, define el plan de implementación paso a paso antes de escribir código. No implementa.
   - **Implementador**: ejecuta el plan del Orquestador. No improvisa ni añade más de lo pedido. Reporta desviaciones al Orquestador.
   - **Revisor**: hace dos pasadas sobre el resultado del Implementador: (a) revisión de errores (TypeScript, linting, tests si aplica), (b) auditoría de calidad (criterios de aceptación del sub-plan, convenciones, sin estilos inline, sin texto hardcodeado, sin deuda técnica obvia). Emite un veredicto: aprobado / requiere corrección con lista de items.

3. **Flujo de trabajo por tarea** (paso a paso, numerado):
   1. Leer la tarea del sub-plan activo.
   2. Orquestador analiza: ¿qué archivos se tocan?, ¿qué skills se necesitan?, ¿hay dependencias entre sub-tareas?
   3. Orquestador produce el plan de implementación (lista de pasos concretos).
   4. Implementador ejecuta el plan paso a paso.
   5. Implementador reporta al Orquestador si encontró algo que el plan no contempló.
   6. Revisor ejecuta revisión de errores: `pnpm build` o `pnpm typecheck` sin errores.
   7. Revisor ejecuta auditoría: verifica los criterios de aceptación del sub-plan.
   8. Si el Revisor aprueba: la tarea se cierra. Si requiere corrección: el Implementador corrige y el Revisor vuelve a auditar.

4. **Cómo invocar cada rol**: comando o instrucción exacta para abrir el agente con el contexto del rol correcto (dependiendo de si se usa Claude Code CLI o OpenCode CLI).

5. **Reglas globales del Harness** (aplican a todos los roles):
   - Nunca hardcodear versiones de paquetes.
   - Nunca estilos inline (`style={{}}`), salvo CSS custom properties con valores de runtime.
   - Nunca texto hardcodeado: todo string vía i18n.
   - Nunca modificar archivos fuera del alcance de la tarea actual.
   - Preferir editar archivos existentes a crear nuevos.
   - Reportar desviaciones antes de implementarlas.

6. **Swap de modelo (Claude → OpenCode)**: instrucción de qué cambiar en el `CLAUDE.md` de cada rol para hacer el swap. Debe ser un cambio de una línea por rol.

7. **Ciclo de prueba** (sección vacía, se rellena en T2.6): documentar el resultado del ciclo de prueba ejecutado al final de F2.

**Criterio de aceptación**: `AGENTS.md` tiene todas las secciones. Se puede leer de corrido y entender exactamente cómo ejecutar un ciclo sin preguntas.

### T2.4 — Crear el rol Implementador

**Archivos**: `.agents/implementer/CLAUDE.md` y `.agents/implementer/context.md`.

**`CLAUDE.md` del Implementador debe contener**:

- **Identidad**: "Eres el Implementador del proyecto Modus Ratio. Tu función es ejecutar el plan producido por el Orquestador, exactamente como está definido, sin añadir ni omitir pasos."
- **Modelo**: `claude-sonnet-4-6`.
- **Responsabilidades explícitas**:
  - Ejecutar cada paso del plan del Orquestador en el orden indicado.
  - Usar las convenciones del proyecto (diseño atómico, barrels, i18n, sin estilos inline).
  - Reportar al Orquestador si un paso del plan es imposible o incorrecto antes de improvisarlo.
  - No añadir lógica o abstracción más allá de lo que la tarea requiere.
- **Lo que NO hace**:
  - No modifica el plan. Si el plan tiene un error, lo escala al Orquestador.
  - No crea archivos fuera del alcance de la tarea.
  - No toma decisiones de diseño no contempladas en el sub-plan o el fundational plan.
  - No añade comentarios en el código salvo que el "por qué" sea no obvio.

**`context.md` del Implementador debe contener**:
- Estructura de carpetas del proyecto (árbol de `src/` y `.agents/`).
- Convenciones de código:
  - Diseño atómico: `atoms → molecules → organisms → templates`.
  - Cada componente en su carpeta: `Componente/Componente.tsx` + `Componente.css` + `index.ts`.
  - Barrels de import/export.
  - Cero estilos inline salvo CSS custom properties con valores de runtime.
  - Cero texto hardcodeado: todo vía i18n.
  - TypeScript estricto, alias `@/` para imports desde `src/`.
  - `pnpm` para toda instalación.
- Referencia a los breakpoints de Tailwind en uso (`sm:270 / md:360 / lg:440 / xl:600`).
- Los criterios de calidad que el Revisor va a verificar (para que el Implementador los conozca de antemano).

**Criterio de aceptación**: el `CLAUDE.md` del Implementador tiene identidad, modelo, responsabilidades y prohibiciones. El `context.md` tiene las convenciones en un formato que permite verificarlas durante la implementación.

### T2.5 — Crear el rol Revisor

**Archivos**: `.agents/reviewer/CLAUDE.md` y `.agents/reviewer/context.md`.

**`CLAUDE.md` del Revisor debe contener**:

- **Identidad**: "Eres el Revisor del proyecto Modus Ratio. Tu función es verificar que lo implementado cumple los criterios de aceptación del sub-plan y las convenciones del proyecto. Haces dos pasadas: revisión de errores y auditoría de calidad."
- **Modelo**: `claude-sonnet-4-6`.
- **Pasada 1 — Revisión de errores**:
  - Ejecutar `pnpm build` (o `pnpm typecheck` si es más rápido para el caso). Cero errores de TypeScript.
  - Verificar que no hay imports rotos.
  - Verificar que los barrels exportan correctamente lo que prometen.
- **Pasada 2 — Auditoría de calidad** (checklist a verificar en cada revisión):
  1. ¿Todos los criterios de aceptación de la tarea del sub-plan están satisfechos?
  2. ¿Hay algún `style={{}}` que no sea una CSS custom property con valor de runtime?
  3. ¿Hay algún texto hardcodeado fuera de los diccionarios de i18n?
  4. ¿Se creó algún archivo fuera del alcance de la tarea?
  5. ¿Se añadió abstracción no pedida (funciones extra, componentes extra, generalizaciones prematuras)?
  6. ¿Se hardcodeó alguna versión de paquete en `package.json` o en un `import`?
  7. ¿Los nombres de archivos, carpetas y exports siguen las convenciones (PascalCase para componentes, camelCase para utils)?
  8. ¿Los componentes React están en sus carpetas propias con `.css` y `index.ts`?
- **Veredicto**: el Revisor emite uno de dos resultados:
  - `APROBADO`: la tarea se cierra.
  - `REQUIERE CORRECCIÓN`: lista numerada de items a corregir, con referencia al criterio violado.
- **Lo que NO hace**: no implementa las correcciones. Las lista y las devuelve al Implementador.

**`context.md` del Revisor**: referencia a los criterios de aceptación del sub-plan activo (se actualiza por fase), al checklist de la Pasada 2 (para no depender solo del `CLAUDE.md`), y a los principios rectores del fundational plan.

**Criterio de aceptación**: el `CLAUDE.md` del Revisor define las dos pasadas, el checklist completo y el formato del veredicto. No hay ambigüedad sobre qué se revisa.

### T2.7 — Documentar el mecanismo de swap Claude → OpenCode

**Objetivo**: que cambiar de Claude a OpenCode por rol sea un cambio en un solo lugar, sin tocar múltiples archivos.

**Diseño del mecanismo**:

Cada `CLAUDE.md` de rol tiene una sección `## Modelo` que contiene:

```
Modelo activo: claude-sonnet-4-6
# Para hacer swap a OpenCode, reemplazar la línea anterior por:
# Modelo activo: opencode/<modelo-equivalente>
```

Eso es todo. OpenCode lee el mismo `CLAUDE.md`. Si OpenCode tiene un parámetro de modelo diferente, documentar el formato exacto en esta sección como comentario.

**En `AGENTS.md`** (sección "Swap de modelo"), documentar:
- Qué línea cambiar en cada `CLAUDE.md` para hacer el swap.
- Si OpenCode requiere algún archivo adicional (p. ej. `opencode.json`) o si solo lee `CLAUDE.md`.
- Confirmar en `F0-hallazgos.md` si OpenCode necesita configuración adicional (este dato lo produce T0.9 de F0).

**Criterio de aceptación**: la sección de swap en `AGENTS.md` es tan clara que hacer el cambio de modelo toma menos de 2 minutos por rol.

### T2.8 — Ciclo de prueba del Harness

**Objetivo**: ejecutar un ciclo completo (Orquestar → Implementar → Revisar → Auditar) sobre una tarea trivial para verificar que el Harness funciona antes de usarlo en F3.

**Tarea de prueba**: "Crear el archivo `CONTRIBUTING.md` en la raíz del repo con las secciones: convenciones de commits, cómo ejecutar el proyecto en local, y referencia al Harness para contribuir."

**Por qué esta tarea**: es pequeña (1 archivo de texto), no tiene dependencias de código, y ejercita el ciclo completo sin riesgo de romper nada.

**Flujo esperado**:
1. Orquestador analiza la tarea, identifica que es solo documentación, planifica: 1 archivo con 3 secciones, sin skills especiales necesarias.
2. Implementador crea `CONTRIBUTING.md` siguiendo el plan.
3. Revisor verifica: ¿cubre las 3 secciones? ¿no hay texto hardcodeado en archivos de código? (no aplica en un `.md`, pero el checklist se ejecuta). Veredicto: APROBADO.
4. El resultado queda documentado en la sección "Ciclo de prueba" de `AGENTS.md`.

**Criterio de aceptación**: `CONTRIBUTING.md` existe con las 3 secciones. El rastro del ciclo (qué decidió el Orquestador, qué hizo el Implementador, qué verificó el Revisor) está documentado en `AGENTS.md`.
