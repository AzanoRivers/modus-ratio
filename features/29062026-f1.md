### T1.1 — Crear el proyecto Astro

**Comando base** (usar la versión exacta confirmada en `F0-hallazgos.md`; no hardcodear versión en el comando):

```
pnpm create astro@latest
```

**Opciones durante el wizard**:
- Plantilla: `Empty` (proyecto vacío, sin plantillas con contenido de ejemplo).
- TypeScript: `Strict`.
- Instalar dependencias: `Yes` (pnpm las gestiona).
- Inicializar git: `Yes`.

**Resultado esperado**: carpeta raíz del proyecto con `astro.config.mjs`, `tsconfig.json`, `src/`, `public/`, `package.json` generado automáticamente.

**Comando no interactivo** (Astro 7 soporta `--yes` y `--skip-houston` para ejecución sin wizard):

```powershell
pnpm create astro@latest . --template minimal --skip-houston --yes --no-git
```

- `.`: inicializar en el directorio actual (ya existe, Astro 7 lo acepta con `--yes`)
- `--template minimal`: proyecto vacío sin contenido de ejemplo
- `--skip-houston`: omite la animación de la mascota (necesario para agentes)
- `--yes`: acepta todos los defaults sin prompts
- `--no-git`: no re-inicializar git (el repo ya existe)

**TypeScript strict**: el flag `--typescript` no existe en Astro 7. El wizard con `--yes` genera `tsconfig.json` con `"extends": "astro/tsconfigs/base"`. Se cambia manualmente a `strict` en T1.3.

**Nota — comportamiento de `astro dev` bajo agente IA**: Astro 7 detecta automáticamente entornos de agente IA (vía librería `am-i-vibing`) y arranca `astro dev` en modo background sin intervención:
- Escribe `.astro/dev.json` con URL, puerto y PID
- Expone `/_astro/status` como health endpoint
- Parar con `astro dev stop`

Para correr el servidor interactivo desde tu propia terminal: `$env:ASTRO_DEV_BACKGROUND=0` antes de `pnpm dev`.

**Criterio de aceptación**: `pnpm dev` arranca sin errores; `/_astro/status` responde 200.

### T1.2 — Agregar integraciones oficiales

**React y Vercel** (comandos automáticos que modifican `astro.config.mjs`):

```powershell
pnpm astro add react
pnpm astro add vercel
```

**Tailwind 4.x** (instalación manual — `pnpm astro add tailwind` instala la versión deprecada `@astrojs/tailwind`, NO usar):

```powershell
pnpm add -D tailwindcss @tailwindcss/vite
```

Luego editar `astro.config.mjs` manualmente para agregar el plugin de Vite:

```js
import { defineConfig } from 'astro/config'
import react from '@astrojs/react'
import vercel from '@astrojs/vercel/serverless'
import tailwindcss from '@tailwindcss/vite'

export default defineConfig({
  output: 'server',
  adapter: vercel(),
  integrations: [react()],
  vite: {
    plugins: [tailwindcss()],
  },
})
```

**Nota**: `pnpm astro add vercel` puede dejar `output: 'hybrid'`. Verificar y corregir a `'server'`.

**Criterio de aceptación**:
- `astro.config.mjs` tiene `output: 'server'`, el adapter de Vercel, React en `integrations`, y `@tailwindcss/vite` en `vite.plugins`.
- `pnpm dev` sigue levantando sin errores.

### T1.3 — Configurar TypeScript estricto

Verificar que `tsconfig.json` incluye `"extends": "astro/tsconfigs/strict"`. Si el wizard lo creó con `base` o `strictest`, cambiarlo a `strict`.

Agregar estas opciones bajo `compilerOptions` si no están:

```json
"baseUrl": ".",
"paths": {
  "@/*": ["src/*"]
}
```

El alias `@/` permite importar desde `src/` sin rutas relativas largas. Se usa en todos los barrels de las fases siguientes.

**Criterio de aceptación**: `pnpm build` no reporta errores de TypeScript.

### T1.4 — Instalar dependencias de negocio

Ejecutar (sin hardcodear versiones, `pnpm` resuelve la última compatible):

```
pnpm add zustand
pnpm add openai
pnpm add @aws-sdk/client-s3 @aws-sdk/s3-request-presigner
pnpm add redis
pnpm add resend
```

**Notas**:
- `@aws-sdk/client-s3` + `@aws-sdk/s3-request-presigner`: son los paquetes para generar las URLs prefirmadas hacia R2 (compatible S3). Ambos se usan solo en servidor (endpoints Astro).
- `redis`: cliente de Redis. Se usa solo en servidor (middleware y endpoints). La versión a instalar es la confirmada en `F0-hallazgos.md`.
- `openai`: SDK oficial de OpenAI. Solo en servidor.
- `resend`: solo en servidor.
- `zustand`: único paquete que también va al cliente (store de estado de la UI).

**Criterio de aceptación**: `package.json` tiene las 6 dependencias agregadas con versiones resueltas (generadas por pnpm, no escritas a mano). `pnpm build` sin errores de módulo faltante.

### T1.6 — Verificar SSR y server-only secrets (sin endpoint de health)

La verificación de que las variables sin `PUBLIC_` no llegan al cliente se hace inspeccionando el bundle generado, no con un endpoint dedicado. Un endpoint de health para Redis no aplica: la conexión a Redis se maneja directamente en cada endpoint que la usa, sin health checks globales.

**Verificación**: abrir DevTools > Network con el dev server activo, cargar la página de prueba, y confirmar que ningún JS descargado contiene el valor de `REDIS_PASSWORD` del `.env.local`.

**Criterio de aceptación**: el valor de `REDIS_PASSWORD` no aparece en ningún bundle del cliente.

### T1.10 — Verificar TypeScript limpio

Ejecutar `pnpm astro check` (no `pnpm build`) para verificar TypeScript:

```powershell
pnpm add -D @astrojs/check typescript   # instalar si no está
pnpm astro check
```

**Nota Windows — `pnpm build` falla localmente**: el adapter `@astrojs/vercel` intenta crear symlinks durante el bundling (`copyDependenciesToFunction`). En Windows, la creación de symlinks requiere Modo Desarrollador activo o privilegios de administrador. La compilación TypeScript + Vite es exitosa; solo falla el paso final de bundling del adapter. Esto no afecta los deploys en Vercel (Linux). Para verificar TS localmente, usar `pnpm astro check`.

**Criterio de aceptación**: `pnpm astro check` reporta `0 errors, 0 warnings, 0 hints`.

## Riesgos

| Riesgo | Mitigación |
|---|---|
| `pnpm astro add vercel` deja `output: 'hybrid'` en vez de `'server'` | Verificar y corregir manualmente después del comando |
| El alias `@/*` no resuelve correctamente en `.astro` y `.tsx` | Confirmar en `tsconfig.json` que `paths` y `baseUrl` están correctos; Astro respeta el `tsconfig` |
| `@aws-sdk` tiene sub-paquetes que no se tree-shake bien en Vercel serverless | En F10 (donde se usa) verificar el bundle size; el impacto en F1 es nulo porque no se importan aún |
| Placeholder de `middleware.ts` vacío causa error si Astro lo valida al arrancar | El export mínimo `export const onRequest = () => {}` evita el error |

## Nota de ejecución

Los archivos placeholder deben exportar algo válido (aunque sea vacío) para que TypeScript no genere errores en `pnpm build`. Regla mínima:

- `*.ts` con función: `export const nombre = () => {}`.
- `*.ts` con objeto: `export const nombre = {}`.
- `*.ts` con interfaz: `export interface Nombre {}`.
- Barrel `index.ts`: `export {}`.

Nada de lógica real en los placeholders: solo la firma mínima que satisface a TypeScript.
