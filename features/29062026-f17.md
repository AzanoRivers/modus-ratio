## Objetivo

Conectar todos los componentes y endpoints ya construidos en un flujo end-to-end coherente. F17 no crea lógica de negocio nueva — orquesta la que ya existe: valida el estado del store antes de cada paso, llama los endpoints en el orden correcto, actualiza el store en cada transición y renderiza la vista correspondiente con una transición VHS entre cada cambio de estado.

## Decisiones ya tomadas

| Decisión | Valor |
|---|---|
| Componente raíz del flujo | `FlowController` — island React con `client:idle`, montado en `index.astro` |
| Transición entre vistas | `CyberTransition` con `transitionKey` basado en la fase; `uploading` y `analyzing` comparten el mismo key `'loading'` para no interrumpir el loader |
| Flujo de reintento tras fallo IA | Si `currentFileKey` existe en el store: saltar presign y upload, ir directo a `/api/analyze`. Si `currentFileKey` es null: `resetForReanalysis()` y volver al formulario |
| Límite de uso | Inline en el formulario (`UsageLimitMessage`), con cuenta regresiva en minutos. No redirige a `/429` — esa página es solo para ban de seguridad de Redis |
| Ban de seguridad | Lo maneja el middleware de F10 (redirige a `/429`). F17 no lo gestiona |
| `store.recordUsage()` | Se llama en el cliente después de recibir 200 de `/api/analyze`, para mantener el espejo de localStorage sincronizado con el servidor |
| Resend alert | Se dispara en el servidor (dentro de `/api/analyze`) cuando el pipeline de IA lanza `AnalysisError`. F17 documenta la actualización del endpoint |
| `resetForReanalysis()` | Nueva acción del store: limpia `currentFile`, `currentFileKey`, `results`, `uploadProgress`, `analysisError` y setea `phase: 'idle'`. No toca `formParams` ni `usageTimestamps` |

## Cierre — Definition of Done

- Flujo completo end-to-end funciona: seleccionar imagen → enviar → loader sube → loader analiza → resultados aparecen con transición VHS.
- Re-análisis ("Analizar otro outfit"): vuelve al formulario con los parámetros previos conservados.
- Reintento tras fallo IA: si `currentFileKey` existe, no re-sube la imagen.
- Con 25 análisis en la última hora: el botón de envío está disabled, `UsageLimitMessage` muestra la cuenta regresiva.
- Un ban de seguridad (middleware de F10) redirige a `/429` — F17 no interfiere.
- Un fallo de IA (502) muestra `ProcessingError` con botón de reintento.
- El servidor envía alerta Resend cuando el pipeline de IA falla.
- Ningún texto hardcodeado.
- Sin `style={{}}` salvo CSS custom properties dinámicas.
- `pnpm build` sin errores TypeScript.

### T17.3 — `src/lib/useAnalysisFlow.ts` — Hook de orquestación

El hook encapsula la lógica asíncrona del flujo completo y expone dos funciones: `startFlow` (flujo completo desde el upload) y `retryFlow` (reintenta solo el análisis).

```ts
import { useAppStore }   from '@/store'
import { uploadToR2 }    from '@/lib/uploadToR2'
import type { FormParams } from '@/components/organisms/HomeForm'

export function useAnalysisFlow() {
  const store = useAppStore()

  async function startFlow(file: File): Promise<void> {
    store.setPhase('uploading')
    store.setCurrentFile(file)
    store.setUploadProgress(0)

    // Paso 1: solicitar URL presignada
    let presignedUrl: string
    let objectKey: string

    try {
      const res = await fetch('/api/upload/presign', {
        method:  'POST',
        headers: { 'Content-Type': 'application/json' },
        body:    JSON.stringify({
          contentType:   file.type,
          contentLength: file.size,
        }),
      })

      if (!res.ok) {
        store.setPhase('error')
        store.setAnalysisError('UPLOAD_PRESIGN_FAILED')
        return
      }

      const json = await res.json()
      presignedUrl = json.data.presignedUrl
      objectKey    = json.data.objectKey
    } catch {
      store.setPhase('error')
      store.setAnalysisError('UPLOAD_PRESIGN_FAILED')
      return
    }

    store.setCurrentFileKey(objectKey)

    // Paso 2: subir a R2
    try {
      await uploadToR2(file, presignedUrl, {
        onProgress: (percent) => store.setUploadProgress(percent),
      })
    } catch {
      store.setPhase('error')
      store.setAnalysisError('UPLOAD_FAILED')
      return
    }

    // Paso 3: analizar
    await runAnalyze(objectKey, store.formParams, store)
  }

  async function retryFlow(): Promise<void> {
    const { currentFileKey, formParams } = useAppStore.getState()

    if (!currentFileKey) {
      // El archivo ya no está en R2 (recarga de página u otro edge case)
      store.resetForReanalysis()
      return
    }

    store.setPhase('analyzing')
    store.setAnalysisError(null)

    await runAnalyze(currentFileKey, formParams, store)
  }

  return { startFlow, retryFlow }
}

// ── Helper interno ──────────────────────────────────────────────────────────

type StoreActions = ReturnType<typeof useAppStore>

async function runAnalyze(
  objectKey:  string,
  formParams: FormParams,
  store:      StoreActions,
): Promise<void> {
  store.setPhase('analyzing')

  let res: Response
  try {
    res = await fetch('/api/analyze', {
      method:  'POST',
      headers: { 'Content-Type': 'application/json' },
      body:    JSON.stringify({ objectKey, formParams }),
    })
  } catch {
    store.setPhase('error')
    store.setAnalysisError('AI_ANALYSIS_FAILED')
    return
  }

  if (res.status === 429) {
    // El servidor detectó límite de uso (posible desincronía con el cliente)
    // Volver a idle — canAnalyze() reflejará el estado real en el próximo ciclo
    store.setPhase('idle')
    return
  }

  if (!res.ok) {
    store.setPhase('error')
    store.setAnalysisError('AI_ANALYSIS_FAILED')
    return
  }

  const json = await res.json()
  store.setResults(json.data)
  store.recordUsage()      // espejo client-side — el servidor ya registró el uso
  store.setPhase('results')
}
```

**Por qué `runAnalyze` es una función auxiliar fuera del hook**: tanto `startFlow` como `retryFlow` necesitan el paso de análisis. Extraerlo evita duplicación y mantiene el hook enfocado en exponer solo `startFlow` y `retryFlow`.

**Por qué `store.recordUsage()` se llama después de recibir 200**: el store client-side es un espejo del servidor. Si se llamara antes (o ante cualquier intento), se consumiría el contador aunque el análisis fallara. El registro autoritativo es el del servidor (Redis).

**Por qué `retryFlow` lee del store con `useAppStore.getState()` en lugar de destructuring**: el `store` que captura el hook via `useAppStore()` tiene el estado en el momento del render. `getState()` garantiza leer el estado actual en el momento de la ejecución del retry, que puede ocurrir un ciclo después.

**Criterio de aceptación**: TypeScript compila sin errores. `startFlow` transiciona correctamente entre fases. Un fallo en presign pone `phase: 'error'`. Un fallo de upload pone `phase: 'error'`. Un 502 de analyze pone `phase: 'error'`. Un 429 vuelve a `phase: 'idle'`. Un 200 pone `phase: 'results'` y llama `recordUsage()`.

### T17.4 — `UsageLimitMessage` (molecule)

Crear `src/components/molecules/UsageLimitMessage/`.

**`UsageLimitMessage.tsx`**:

```tsx
import './UsageLimitMessage.css'
import { useEffect, useState } from 'react'
import { useAppStore }         from '@/store'
import type { Translations }   from '@/i18n/types'

interface UsageLimitMessageProps {
  t: Translations['home']
}

export function UsageLimitMessage({ t }: UsageLimitMessageProps) {
  const resetAt = useAppStore((s) => s.resetAt)
  const [minutesLeft, setMinutesLeft] = useState<number | null>(calcMinutes(resetAt()))

  useEffect(() => {
    const interval = setInterval(() => {
      setMinutesLeft(calcMinutes(resetAt()))
    }, 30_000)
    return () => clearInterval(interval)
  }, [resetAt])

  return (
    <div className="usage-limit-message" role="alert">
      <p className="usage-limit-message__title">{t.usageLimitTitle}</p>
      <p className="usage-limit-message__body">{t.usageLimitBody}</p>
      {minutesLeft !== null && minutesLeft > 0 && (
        <p className="usage-limit-message__countdown">
          {t.usageLimitResetsAt}{' '}
          <span className="usage-limit-message__minutes">
            {minutesLeft} {t.usageLimitMinutes}
          </span>
        </p>
      )}
    </div>
  )
}

function calcMinutes(resetAtMs: number | null): number | null {
  if (!resetAtMs) return null
  const diff = resetAtMs - Date.now()
  return diff <= 0 ? null : Math.ceil(diff / 60_000)
}
```

### T17.6 — `FlowController` (organism)

Crear `src/components/organisms/FlowController/`.

**`FlowController.tsx`**:

```tsx
import './FlowController.css'
import { useAppStore }      from '@/store'
import { useAnalysisFlow }  from '@/lib/useAnalysisFlow'
import { CyberTransition }  from '@/components/atoms/CyberTransition'
import { AnalysisLoader }   from '@/components/organisms/AnalysisLoader'
import { ResultsPanel }     from '@/components/organisms/ResultsPanel'
import { HomeForm }         from '@/components/organisms/HomeForm'
import { UsageLimitMessage } from '@/components/molecules/UsageLimitMessage'
import { ProcessingError }   from '@/components/molecules/ProcessingError'
import type { Translations } from '@/i18n/types'

interface FlowControllerProps {
  t: Translations
}

export function FlowController({ t }: FlowControllerProps) {
  const phase      = useAppStore((s) => s.phase)
  const canAnalyze = useAppStore((s) => s.canAnalyze)

  const { startFlow, retryFlow } = useAnalysisFlow()

  // Clave de transición: uploading y analyzing comparten 'loading'
  // para no interrumpir el loader al pasar de una fase a la otra
  const transitionKey =
    phase === 'uploading' || phase === 'analyzing' ? 'loading' : phase

  const handleSubmit = () => {
    const file = useAppStore.getState().currentFile
    if (!file || !canAnalyze()) return
    startFlow(file)
  }

  const handleReanalyze = () => {
    useAppStore.getState().resetForReanalysis()
  }

  const handleRetry = () => {
    const { currentFileKey } = useAppStore.getState()
    if (currentFileKey) {
      retryFlow()
    } else {
      useAppStore.getState().resetForReanalysis()
    }
  }

  return (
    <div className="flow-controller">
      <CyberTransition transitionKey={transitionKey} effect="scanIn">

        {phase === 'idle' && (
          <div className="flow-controller__idle">
            {!canAnalyze() && <UsageLimitMessage t={t.home} />}
            <HomeForm
              t={t}
              onSubmit={handleSubmit}
              disabled={!canAnalyze()}
            />
          </div>
        )}

        {(phase === 'uploading' || phase === 'analyzing') && (
          <AnalysisLoader t={t.loader} />
        )}

        {phase === 'results' && (
          <ResultsPanel t={t} onReanalyze={handleReanalyze} />
        )}

        {phase === 'error' && (
          <ProcessingError
            t={t.analysis.processingError}
            onRetry={handleRetry}
          />
        )}

      </CyberTransition>
    </div>
  )
}
```

**Notas de implementación**:

- `useAppStore.getState()` en los handlers: los handlers son funciones llamadas en eventos de usuario (onClick), no durante el render. Leer el estado en el momento del evento con `getState()` garantiza valores actuales. Usando `useAppStore((s) => s.currentFile)` en el render y leyéndolo en el handler funcionaría también, pero leer con `getState()` es más explícito y evita que el componente re-renderice por cambios en `currentFile`.
- El `disabled` que se pasa a `HomeForm` es la suma de dos condiciones: `!canAnalyze()`. El botón también estará disabled si no hay archivo seleccionado — eso lo controla `HomeForm` internamente con su validación de `currentFile !== null` (F7/F8).
- `FlowController` no gestiona el ban de seguridad: si el middleware de F10 banea la IP, redirige a `/429` antes de que este componente haga cualquier request. F17 no necesita detectar ni manejar ese caso.

**Criterio de aceptación**: TypeScript acepta todos los tipos. Cambiar `phase` en el store produce la transición VHS al componente correcto. `handleSubmit` solo se ejecuta si hay `currentFile` y `canAnalyze()` es true.

Editar `src/components/organisms/HomeForm/HomeForm.tsx`.

Actualmente `HomeForm` tiene:
```ts
interface HomeFormProps {
  t: Translations
}
```

F17 añade dos props:
```ts
interface HomeFormProps {
  t:         Translations
  onSubmit:  () => void
  disabled:  boolean
}
```

**Cambios en el botón de envío**:
```tsx
<Button
  type="button"
  disabled={disabled || !currentFile || !style}
  onClick={onSubmit}
>
  {t.home.analyzeButton}
</Button>
```

Donde `currentFile` y `style` se leen del store (ya migrado en F8, T8.4).

**Por qué `type="button"` y no `type="submit"`**: `HomeForm` no es un `<form>` HTML con submit nativo — es un conjunto de controles React que acumulan estado en el store. El botón llama `onSubmit` directamente, no dispara un evento de formulario.

**Criterio de aceptación**: con `disabled={true}`, el botón no puede pulsarse. Con `disabled={false}` y sin archivo seleccionado, el botón sigue disabled (segunda condición interna). Con archivo y parámetros válidos, el click llama `onSubmit`.

## Riesgos

| Riesgo | Mitigación |
|---|---|
| `CyberTransition` remonta sus hijos en cada cambio de `transitionKey` — si React hace batching de dos cambios de fase seguidos, la transición puede quedar inconsistente | Verificar en QA que `uploading → analyzing` no produce un flash o doble transición. Si ocurre: usar `useEffect` para detectar el cambio de phase y aplicar la transitionKey con un delay de un frame |
| `HomeForm` usa `useState` local para el archivo en F7 pero F8 T8.4 migra el archivo al store con `currentFile` — verificar que T8.4 de F8 se ejecutó correctamente antes de que F17 lea `currentFile` del store | El Orquestador confirma que F8 está cerrado y que `currentFile` vive en el store antes de que el Implementador empiece T17.6 |
| `uploadToR2` (F11) retorna `Promise<void>` y lanza en caso de error — el try/catch en `startFlow` captura la excepción, pero si `uploadToR2` tiene un error interno que no lanza sino que resuelve silenciosamente, el flow no detecta el fallo | Verificar en F11 que `uploadToR2` siempre lanza en caso de fallo de red o de respuesta no-2xx de R2. Si resuelve silenciosamente en error: actualizar F11 para que lance, o agregar detección en `startFlow` |
| `sendAlertEmail` no existe con ese nombre en `resend.ts` de F10 | El Implementador lee `src/lib/resend.ts` antes de escribir la llamada. Adaptar el nombre y firma según lo que exporte F10 |
| Latencia de `client:idle` — la isla se hidrata tarde y el usuario intenta interactuar antes de que React esté listo | Si ocurre en QA: cambiar a `client:load` en `index.astro`. El impacto en performance inicial es mínimo dado el tamaño del bundle |
