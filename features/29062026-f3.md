## Objetivo

Producir el sistema de i18n que todas las fases siguientes consumen. El sistema es tipado, detectado en servidor y opaco para el cliente: ningún componente sabe qué idioma está activo; solo recibe el diccionario correcto. Una vez terminada esta fase, escribir cualquier string directamente en un componente es un error de convención, detectable con una búsqueda de texto.

### T3.4 — Implementar `getDictionary` y tipo `Locale` (index.ts)

`src/i18n/index.ts` exporta:

```ts
export type Locale = 'es' | 'en'

export function getDictionary(locale: Locale): Translations {
  return locale === 'es' ? es : en
}
```

Y re-exporta `Translations` desde `types.ts` para que los componentes importen desde un solo punto:

```ts
export type { Translations } from './types'
```

**Por qué sincrónico**: los diccionarios son módulos TS estáticos cargados en tiempo de build. No hay latencia, no hay `await`. Una función sincrónica es más simple y no introduce manejo de promesas en los componentes.

**Criterio de aceptación**: `getDictionary('es')` retorna el objeto español, `getDictionary('en')` el inglés. TypeScript infiere correctamente el tipo de retorno como `Translations`.

### T3.5 — Implementar `getLocale` (utils/getLocale.ts)

Util server-side que lee el header `accept-language` de la request y aplica la regla de detección.

**Firma**:

```ts
export function getLocale(request: Request): Locale
```

**Lógica** (sin librerías externas, sin parsers de Accept-Language):
1. Leer `request.headers.get('accept-language')`.
2. Si no existe o está vacío: retornar `'en'`.
3. Tomar el primer tag de la lista (separar por `,`, tomar el primero; luego separar por `;` para quitar el q-value, tomar la parte del tag).
4. Pasar a minúsculas y verificar si empieza por `'es'`: si sí → `'es'`; si no → `'en'`.

**Ejemplo de headers que devuelven `'es'`**: `es`, `es-ES`, `es-419`, `es-MX;q=0.9`, `es-US,en;q=0.8`.
**Ejemplo de headers que devuelven `'en'`**: `en-US`, `fr-FR`, `pt-BR`, `de`, ausencia de header.

**Referencia**: patrón de `00_AzanoRivers/src/utils/getLang.ts`, adaptado para la regla ES/EN de Modus Ratio (en AzanoRivers el fallback era español; aquí el fallback es inglés).

**Criterio de aceptación**:
- `getLocale` solo se usa en código de servidor (endpoints, layouts, middleware). No se importa en componentes React.
- Los 6 ejemplos de arriba producen el resultado correcto.
- No tiene dependencias de paquetes externos.

### T3.6 — Declarar `App.Locals.language` en TypeScript

Astro expone `Astro.locals` con un tipo extensible vía `App.Locals`. Para que TypeScript sepa que `Astro.locals.language` existe y es de tipo `Locale`, se declara en `src/env.d.ts`:

```ts
declare namespace App {
  interface Locals {
    language: import('@/i18n').Locale
  }
}
```

**Por qué ahora y no en F10**: el middleware que puebla `Astro.locals.language` llega en F10, pero si la declaración no existe antes, cualquier archivo `.astro` que acceda a `Astro.locals.language` entre F3 y F10 generará un error de TypeScript. Declararlo en F3 evita ese problema.

**Criterio de aceptación**: `Astro.locals.language` es de tipo `Locale` en cualquier archivo `.astro` sin necesidad de castear.

### T3.7 — Documentar y demostrar el patrón de entrega a islas React

Este patrón es la convención que todos los componentes React hidratados deben seguir desde F6 en adelante. Se documenta aquí y se demuestra en la página de demo de T3.8.

**El patrón**:

En el archivo `.astro` (servidor):
```astro
---
import { getLocale, getDictionary } from '@/i18n'
import MiComponente from '@/components/atoms/MiComponente'

const locale = Astro.locals.language ?? getLocale(Astro.request)
const t = getDictionary(locale)
---

<MiComponente t={t} client:idle />
```

En el componente React (`MiComponente.tsx`):
```tsx
import type { Translations } from '@/i18n'

interface Props {
  t: Translations
}

export function MiComponente({ t }: Props) {
  return <p>{t.home.title}</p>
}
```

**Reglas del patrón**:
1. El `.astro` siempre resuelve `locale` antes de renderizar (via `Astro.locals.language` si el middleware lo puso, o via `getLocale` como fallback mientras F10 no está).
2. El componente React recibe `t: Translations` como prop. Nunca llama `getDictionary` internamente ni hace fetch del idioma.
3. El tipo de `t` siempre es `Translations` (tipado, no `any`, no `Record<string, string>`).
4. Los sub-namespaces pueden desestructurarse en el componente: `const { home, form } = t`.
5. Para textos con variables dinámicas: usar `.replace('{key}', value)` directamente en el template.

**Por qué no pasar solo `locale` y dejar que el componente importe el diccionario**: si el componente hace `import { getDictionary } from '@/i18n'`, Vite incluirá ambos diccionarios (`es.ts` y `en.ts`) en el bundle del cliente porque el import es dinámico. Al pasar `t` serializado como prop, Astro solo serializa el diccionario del idioma activo.

**Criterio de aceptación**: la página demo (T3.8) implementa este patrón. Está documentado en `AGENTS.md` o en el `context.md` del Implementador para que todas las fases lo sigan.

## Riesgos

| Riesgo | Mitigación |
|---|---|
| Algún namespace de `Translations` queda incompleto al implementar `es.ts` o `en.ts` | TypeScript lo detecta al compilar; `pnpm build` falla con el campo faltante |
| Clave con variable dinámica (`{time}`) mal documentada causa inconsistencias entre idiomas | La interfaz fuerza que ambos diccionarios tengan la misma clave; el formato del placeholder se documenta en el comentario de la clave en `types.ts` |
| El Implementador importa `getDictionary` en un componente React | El Revisor tiene este punto en su checklist de auditoría; el Orquestador lo bloquea en la planificación |
| `accept-language` en algunos navegadores mobile tiene formato inusual | La lógica de `getLocale` toma solo el primer tag y lo procesa con `toLowerCase().startsWith('es')`, lo cual cubre todos los formatos estándar de BCP 47 |
| `App.Locals` declarado en `env.d.ts` no se carga si Astro no sabe del archivo | Verificar que `tsconfig.json` incluye `src/env.d.ts` en `include`; Astro lo carga automáticamente si está en `src/` |

## Convención de claves (para fases siguientes)

Cuando una fase posterior necesite agregar claves al diccionario:
1. Agregar la clave en `types.ts` (dentro del namespace correcto).
2. Agregar el string correspondiente en `es.ts`.
3. Agregar el string correspondiente en `en.ts`.
4. `pnpm build` confirma que los 3 archivos son consistentes.
5. Nunca saltarse el paso 1: ningún diccionario puede tener claves que no estén en `Translations`.
