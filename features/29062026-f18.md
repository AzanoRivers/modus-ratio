## Bug post-cierre (01-07-2026): miniatura vacía al seleccionar HEIC

El usuario probó con una imagen `.heic` real y la miniatura del `Dropzone` quedaba vacía. Causa: `ensureJpeg` (T18.9) solo se llamaba en `useAnalysisFlow.ts` justo antes de subir a R2, no al seleccionar el archivo. `Dropzone.tsx` generaba la vista previa con `URL.createObjectURL(file)` sobre el archivo HEIC original, y ningún navegador salvo Safari puede decodificar HEIC dentro de una etiqueta `<img>`.

**Fix**: `Dropzone.tsx` ahora llama `ensureJpeg` al seleccionar el archivo (antes de generar la miniatura), muestra un estado "Preparando tu foto..." con ícono girando mientras convierte, y pasa el archivo **ya convertido** a `onFileSelected` (no el HEIC original). Esto significa que `currentFile` en el store ya es JPEG desde el momento de la selección, así que la llamada a `ensureJpeg` en `useAnalysisFlow.ts` (T18.10) se vuelve un passthrough sin costo para estos archivos (ya no es HEIC), no una doble conversión real.

**Segundo hallazgo, mismo día**: probado con un HEIC real de iPhone, `heic2any` falló al decodificarlo (`Could not parse HEIF file`, error interno de la librería `libheif.js` que envuelve, no un bug de integración nuestro). El catch original de T18.10 hacía fallback silencioso al archivo HEIC sin convertir, que el navegador tampoco puede mostrar: la miniatura quedaba vacía sin ninguna explicación para el usuario. **No todas las variantes de HEIC son decodificables por `heic2any`**, es una limitación real de la librería, no algo que se pueda garantizar al 100%. Corregido: ahora un fallo de conversión se trata como un error de validación más (nuevo estado `'heic'` en `Dropzone.tsx`, junto a `'format'`/`'size'`), con mensaje explícito (`t.errors.imageHeicFailed`: "no pudimos procesar esta foto HEIC, prueba exportarla como JPG o toma otra foto") y `onFileSelected(null)` para no dejar avanzar con un archivo que el pipeline de IA tampoco podría procesar (OpenAI vision no acepta HEIC crudo).

### T18.1 — Verificar y corregir `overflow-x` en `globals.css`

**Qué hacer**: abrir `src/styles/globals.css` y buscar cualquier `overflow-x: clip` en `html`, `body` o `*`. Si existe, reemplazar por `overflow-x: hidden`.


### T18.3 — Añadir `-webkit-clip-path` en las animaciones de F5

Esta es la corrección más crítica. `@keyframes cyberCRTEnter` (definido en F5, `globals.css`) usa `clip-path: inset(...)` en cada frame. En iOS Safari antiguo, `clip-path` animado puede fallar sin prefijo.

**Localizar en `src/styles/globals.css`**:

```css
@keyframes cyberCRTEnter {
  0%   { clip-path: inset(100dvh 0 0 0); filter: brightness(3.5) saturate(0) hue-rotate(90deg); }
  20%  { clip-path: inset(65dvh 0 0 0);  filter: brightness(3) saturate(0.1) hue-rotate(40deg); }
  /* ... más frames ... */
}
```

**Añadir `-webkit-clip-path` en cada frame**:

```css
@keyframes cyberCRTEnter {
  0%   {
    -webkit-clip-path: inset(100dvh 0 0 0);
    clip-path: inset(100dvh 0 0 0);
    filter: brightness(3.5) saturate(0) hue-rotate(90deg);
  }
  20%  {
    -webkit-clip-path: inset(65dvh 0 0 0);
    clip-path: inset(65dvh 0 0 0);
    filter: brightness(3) saturate(0.1) hue-rotate(40deg);
  }
  /* idem para cada frame */
}
```

Lo mismo aplica para `@keyframes cyberCRTEnterMobile` y para la clase `.page-enter` si tiene `clip-path` directamente (no solo en keyframes):

```css
.page-enter {
  -webkit-clip-path: inset(0 0 0 0);   /* si hay clip-path inicial */
  clip-path: inset(0 0 0 0);
}
```

Y para `@keyframes ctScanIn` en `CyberTransition.css` (F5), que también usa `clip-path`:

```css
@keyframes ctScanIn {
  from {
    -webkit-clip-path: inset(0 0 100% 0);
    clip-path: inset(0 0 100% 0);
  }
  to {
    -webkit-clip-path: inset(0 0 0% 0);
    clip-path: inset(0 0 0% 0);
  }
}
```

**Criterio de aceptación**: todo `clip-path:` dentro de bloques `@keyframes` o en selectores que se animan tiene `-webkit-clip-path:` con el mismo valor inmediatamente antes.

### T18.9 — `src/lib/ensureJpeg.ts` — Utilidad de conversión HEIC

```ts
export async function ensureJpeg(file: File): Promise<File> {
  if (!isHeic(file)) return file

  const { default: heic2any } = await import('heic2any')

  const result = await heic2any({
    blob:    file,
    toType:  'image/jpeg',
    quality: 0.92,
  })

  const blob     = Array.isArray(result) ? result[0] : result
  const baseName = file.name.replace(/\.(heic|heif)$/i, '')

  return new File([blob], `${baseName}.jpg`, { type: 'image/jpeg' })
}

function isHeic(file: File): boolean {
  if (file.type === 'image/heic' || file.type === 'image/heif') return true
  // Fallback por extensión: iOS a veces reporta type vacío en archivos HEIC
  const ext = file.name.split('.').pop()?.toLowerCase()
  return ext === 'heic' || ext === 'heif'
}
```

**Por qué dynamic import**: `heic2any` incluye un módulo WASM de ~900KB. Cargarlo en el bundle inicial aumenta el LCP innecesariamente para usuarios que no tienen iOS o no usan HEIC. El dynamic import lo descarga solo cuando `isHeic(file)` es `true`.

**Por qué `quality: 0.92`**: balance entre calidad visual (suficiente para que GPT-4o-mini analice bien el outfit) y tamaño de archivo. A 0.92 el JPEG es visualmente indistinguible del HEIC original pero ocupa menos.

**Por qué el fallback por extensión**: en algunos dispositivos iOS (especialmente versiones anteriores o con configuración de cámara específica), el `file.type` puede llegar vacío aunque el archivo sea HEIC. Verificar la extensión cubre este caso.

**Por qué no convertir a WebP**: WebP tiene mejor compresión, pero `heic2any` soporta JPEG y PNG como targets. JPEG es más seguro para la compatibilidad con el endpoint de visión de OpenAI.

**Criterio de aceptación**:
- `ensureJpeg(jpegFile)` devuelve el mismo `File` sin modificar.
- `ensureJpeg(heicFile)` devuelve un nuevo `File` con `type: 'image/jpeg'` y extensión `.jpg`.
- Un archivo con `type: ''` y nombre `photo.HEIC` pasa el check por extensión y se convierte.
- TypeScript compila sin errores.

### T18.10 — Actualizar `useAnalysisFlow.ts`

Editar `src/lib/useAnalysisFlow.ts` para llamar `ensureJpeg` antes del presign.

Localizar el inicio de `startFlow`:

```ts
// Antes (F17):
async function startFlow(file: File): Promise<void> {
  store.setPhase('uploading')
  store.setCurrentFile(file)
  store.setUploadProgress(0)

  // Paso 1: solicitar URL presignada
  ...
```

**Después (F18)**:

```ts
async function startFlow(file: File): Promise<void> {
  store.setPhase('uploading')
  store.setUploadProgress(0)

  // Convertir HEIC → JPEG antes de hacer nada (transparente para el usuario)
  let fileToUpload: File
  try {
    fileToUpload = await ensureJpeg(file)
  } catch {
    // heic2any falló — intentar continuar con el archivo original
    // El servidor puede rechazarlo si HEIC no es soportado, pero es mejor intentarlo
    fileToUpload = file
  }

  store.setCurrentFile(fileToUpload)

  // Paso 1: solicitar URL presignada (con el tipo del archivo convertido)
  ...
```

**El resto de `startFlow` usa `fileToUpload` en lugar de `file` para el presign body y el upload**.

**Por qué catch silencioso en la conversión**: si `heic2any` falla (WASM no carga, archivo corrupto), el flujo no debe bloquearse. Se intenta subir el archivo original — el servidor puede rechazarlo (415) o aceptarlo. El error de conversión no es crítico para el flujo.

**Criterio de aceptación**: subir un archivo HEIC llama `ensureJpeg`. El presign body envía `contentType: 'image/jpeg'` y `contentLength` del archivo convertido. El archivo original (HEIC) no viaja a R2.
