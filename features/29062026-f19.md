### T19.1 — Crear `vercel.json`

Crear en la raíz del proyecto `vercel.json`:

```json
{
  "functions": {
    "src/pages/api/analyze.ts": {
      "maxDuration": 60
    }
  }
}
```

**Por qué 60s**: el pipeline de IA (GPT-4o-mini + Minimax M3) dura ~8-10s en condiciones normales. Con latencia de red en producción puede llegar a 15-20s. El límite de 10s del plan Free Vercel es insuficiente. Con 60s hay margen amplio y es el máximo del plan Pro.

**Por qué solo `/api/analyze`**: `/api/upload/presign` es una operación ligera (<1s). Configurar un timeout alto innecesariamente no aporta nada.

**Si el plan de Vercel es Free (10s)**: el pipeline fallará con timeout. Opciones:
- Subir al plan Pro antes de desplegar.
- Separar el pipeline en dos endpoints llamados en secuencia desde el cliente: `/api/analyze/extract` (GPT, ~3s) y `/api/analyze/score` (Minimax, ~5s), cada uno dentro de 10s. Documentar como plan B en `F19-hallazgos.md` si es necesario.

**Criterio de aceptación**: `vercel.json` existe en la raíz. `pnpm build` no lo rechaza. El deploy de Vercel acepta la configuración.

### T19.4 — Ejecutar `pnpm setup:r2-lifecycle` contra producción

El script de F14 configura la lifecycle rule en el bucket de R2. Ejecutar contra el bucket de producción:

```powershell
pnpm setup:r2-lifecycle
```

El script lee las credenciales del `.env` local (que apuntan al bucket de producción, o configurado explícitamente para este paso).

**Verificar en R2 Dashboard** después de ejecutar:
- Cloudflare Dashboard → R2 → [bucket] → Settings → Lifecycle Rules
- Debe aparecer la regla `cleanup-uploads` con:
  - Status: Enabled
  - Prefix: `uploads/`
  - Expiration: 1 day after object creation

**Si el bucket ya tenía lifecycle rules de otro proyecto**: el script de F14 advierte sobre esto. Revisar el código del script antes de ejecutar — asegurarse de que incluye las reglas existentes en el array `Rules` antes de sobrescribir.

**Criterio de aceptación**: la regla aparece en R2 Dashboard con los parámetros correctos. El script termina sin error.

### T19.6 — Configurar subdominio `modus.azanolabs.com`

**Paso 1 — Añadir dominio en Vercel**:

Vercel Dashboard → proyecto → Settings → Domains → Add → `modus.azanolabs.com`.

Vercel muestra el valor CNAME que hay que apuntar.

**Paso 2 — Configurar DNS en Cloudflare**:

Cloudflare Dashboard → `azanolabs.com` → DNS → Add Record:

```
Type:    CNAME
Name:    modus
Target:  cname.vercel-dns.com   (o el valor exacto que indica Vercel)
Proxy:   DNS only (nube gris) — para que Vercel gestione el TLS
TTL:     Auto
```

**Nota sobre Cloudflare Proxy**: si se activa el proxy de Cloudflare (nube naranja), el TLS lo gestiona Cloudflare en lugar de Vercel. Esto puede interferir con el certificado de Vercel. Empezar con "DNS only" y activar el proxy después si se desea (Vercel y Cloudflare son compatibles en modo proxy, pero requiere configuración adicional).

**Paso 3 — Verificar SSL en Vercel**:

Vercel Dashboard → proyecto → Settings → Domains → `modus.azanolabs.com` → verificar que muestra "Valid Configuration" con certificado SSL activo. Puede tardar hasta 10 minutos.

**Criterio de aceptación**: `https://modus.azanolabs.com` carga la app con certificado SSL válido. No hay warnings de seguridad en el browser.

### T19.8 — Verificaciones de seguridad y comportamiento del sistema

**Verificación 1: imagen nunca cruza Vercel**

1. Abrir la app en producción.
2. Abrir Chrome DevTools → Network tab.
3. Subir una imagen.
4. Verificar que el PUT va directamente a `*.r2.cloudflarestorage.com` o al dominio del bucket R2.
5. Verificar que ninguna request va a `modus.azanolabs.com` (Vercel) con el body binario de la imagen.

**Verificación 2: objeto R2 eliminado después del análisis**

1. Anotar el `objectKey` que aparece en la request de presign (Network tab).
2. Completar el flujo de análisis hasta recibir resultados.
3. Intentar acceder a `${R2_PUBLIC_URL}/${objectKey}` en el browser.
4. Debe responder 403 o 404 — el objeto ya no existe.

**Verificación 3: rate limit Redis funciona**

1. Hacer 3 análisis reales en producción.
2. Verificar en Redis (usando `redis-cli` o Upstash Dashboard si aplica):
   ```
   ZCARD modusratio:usage:<tu-IP>
   ```
   Debe devolver 3.
3. El `resetAt()` del store muestra el tiempo correcto.

**Verificación 4: límite de uso client-side bloquea correctamente**

1. Abrir DevTools → Application → LocalStorage → `modus-ratio-store`.
2. Editar manualmente `usageTimestamps` para incluir 25 timestamps del último minuto.
3. Recargar la página.
4. El botón de análisis debe estar disabled y `UsageLimitMessage` visible.

**Verificación 5: páginas de error responden**

- `https://modus.azanolabs.com/ruta-inexistente` → responde 404 con la página de error correcta.
- `https://modus.azanolabs.com/429` → responde 429 con la página correcta.

**Verificación 6: middleware redirige a `/429` ante ban**

Difícil de probar en producción sin generar un ban real. Documentar como "verificado en local" con el middleware de F10 activo.

**Criterio de aceptación**: todas las verificaciones pasan. Los resultados se registran en `F19-smoke-test.md`.

## Riesgos

| Riesgo | Mitigación |
|---|---|
| Plan Vercel Free (10s timeout) — el pipeline de IA supera el límite | Verificar el plan antes del deploy (T19.1). Si es Free: implementar el plan B (dos endpoints en secuencia) antes de hacer el deploy |
| CNAME de Cloudflare con proxy activo interfiere con el TLS de Vercel | Empezar con proxy desactivado ("DNS only"). Activar el proxy Cloudflare después de confirmar que el SSL de Vercel funciona |
| Variables de entorno faltantes en producción causan error 500 al iniciar | Verificar `env.ts` contra la lista de variables antes del primer deploy (T19.2). Revisar los logs del deploy si falla |
| R2 lifecycle rule sobreescribe reglas existentes del bucket | El script de F14 advierte sobre esto. Verificar manualmente el bucket antes de ejecutar el script |
| Objeto R2 no se elimina después del análisis (deleteR2Object fail-open) | El lifecycle rule de 24h es la red de seguridad. Verificar en la verificación 2 (T19.8). Si el objeto persiste más de 24h después del análisis: revisar el lifecycle rule |
| El smoke test falla por latencia de red real mayor a la estimada | Si el análisis tarda > 30s: investigar qué modelo es el cuello de botella (logs de Vercel). Si tarda > 60s: subir el `maxDuration` o implementar plan B de dos endpoints |
| `F19-smoke-test.md` no cubre HEIC por falta de iPhone | Documentar como pendiente y verificar en la primera sesión con dispositivo iOS disponible. No bloquea el cierre de F19 si todo lo demás pasa |
