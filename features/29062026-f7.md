### T7.2 — Organismo `HomeForm` con estado local React

El organismo `HomeForm` es la isla React que contiene todo el formulario interactivo. En F7 usa `useState` local. En F8 se migra a Zustand, por lo que los nombres de los campos de estado deben ser los definitivos desde ahora.

**Props**:
```ts
interface HomeFormProps {
  t: Translations
}
```

**Estado interno** (nombres definitivos para que F8 no los renombre):
```ts
interface FormState {
  file:       File | null
  style:      StyleOption | null
  height:     number | null        // cm
  skinColor:  SkinColor | null
  build:      BuildValue
  gender:     boolean              // checkbox de género activado
}

const initialState: FormState = {
  file:      null,
  style:     null,
  height:    null,
  skinColor: null,
  build:     { upper: null, lower: null },
  gender:    false,
}
```

**Por qué los nombres definitivos ya en F7**: F8 crea un store Zustand con exactamente estos campos y migra el `useState` a `useStore`. Si los nombres cambian en F8, hay que actualizar referencias en todo `HomeForm`. Establecerlos ahora evita ese refactor.

**Estructura del componente**:

```tsx
export function HomeForm({ t }: HomeFormProps) {
  const [form, setForm] = useState<FormState>(initialState)

  function update<K extends keyof FormState>(key: K, value: FormState[K]) {
    setForm(prev => ({ ...prev, [key]: value }))
  }

  return (
    <div className="home-form__navigator">
      <div className="home-form__nav-bar">
        {/* Barra decorativa del "terminal" */}
      </div>
      <div className="home-form__body">
        <Dropzone t={t} onFileSelected={f => update('file', f)} />
        <StyleSelector t={t} value={form.style} onChange={v => update('style', v)} />
        <section className="home-form__optional">
          <HeightSlider t={t} value={form.height} onChange={v => update('height', v)} />
          <SkinColorSelector t={t} value={form.skinColor} onChange={v => update('skinColor', v)} />
          <BuildSelector t={t} value={form.build} onChange={v => update('build', v)} />
          <GenderCheckbox t={t} checked={form.gender} onChange={v => update('gender', v)} />
        </section>
        <Button
          variant="primary"
          disabled={!form.file || !form.style}
          onClick={() => { /* F17 */ }}
        >
          {t.home.analyzeButton}
        </Button>
      </div>
    </div>
  )
}
```

**Nota sobre el `Button`**: en F7 el `onClick` está vacío. El botón se deshabilita si no hay archivo o estilo seleccionado (los dos campos requeridos). Este comportamiento es definitivo: se mantiene igual en F17 cuando el handler real se conecta.

**Criterio de aceptación**: los 6 campos del formulario actualizan el estado correctamente. El botón está disabled cuando `file === null` o `style === null`. No hay errores de consola en DevTools.

### T7.3 — CSS del organismo `HomeForm`

El "formulario tipo navegador" es visualmente un terminal o ventana de browser con una barra superior decorativa, borde neón y fondo translúcido oscuro.

**Estética del "navegador"** (inspirada en el blueprint cyberpunk):
- Contenedor principal: borde `var(--neon-blue)` con glow, border-radius, `background: rgba(4, 21, 40, 0.75)`, `backdrop-filter: blur(8px)`
- Barra superior (`.home-form__nav-bar`): franja de ~32px de alto, fondo ligeramente más claro, borde inferior neón, tres círculos decorativos a la izquierda (verde/amarillo/rojo, como un terminal macOS) — solo decorativos, sin función
- Cuerpo del formulario (`.home-form__body`): padding interno, gap entre secciones

**Clases y estados**:

```
.home-form__navigator        ← contenedor principal (borde + fondo)
.home-form__nav-bar          ← barra superior decorativa
.home-form__nav-dots         ← los 3 círculos decorativos
.home-form__nav-dot--red     ← círculo rojo decorativo
.home-form__nav-dot--yellow  ← círculo amarillo decorativo
.home-form__nav-dot--green   ← círculo verde decorativo
.home-form__body             ← área de contenido del formulario
.home-form__optional         ← sección de campos opcionales
.home-form__optional-title   ← "Parámetros opcionales" (si se incluye un label de sección)
```

**Responsive** (la distribución de los campos opcionales cambia por breakpoint):

| Breakpoint | Layout de campos opcionales |
|---|---|
| sm (270px) | Columna única: HeightSlider → SkinColorSelector → BuildSelector → GenderCheckbox |
| md (360px) | Igual que sm pero con más espacio horizontal |
| lg (440px) | Igual que sm |
| xl (600px) | Dos columnas: [HeightSlider / SkinColorSelector] en una fila, [BuildSelector / GenderCheckbox] en otra |

En xl, la sección `.home-form__optional` usa `display: grid; grid-template-columns: 1fr 1fr; gap: ...`.

**Criterio de aceptación**: el "terminal" es visualmente reconocible con la barra superior y los puntos decorativos. A 600px los campos opcionales están en 2 columnas. A 270px están en columna única sin desbordamiento.

### T7.5 — Directiva de hidratación para `HomeForm`

**Directiva elegida: `client:idle`**

**Justificación**:
- `client:idle`: el browser hidrata el componente cuando tiene ciclos de CPU libre (generalmente < 300ms después del paint inicial). El usuario ve el formulario renderizado estáticamente y en < 300ms se vuelve interactivo. Es el equilibrio entre rendimiento y experiencia.
- `client:load`: hidrata inmediatamente al cargar el JS del componente. Es más agresivo y bloquea el hilo principal. No necesario para un formulario que el usuario tardará unos segundos en rellenar.
- `client:visible`: hidrata cuando el elemento entra en el viewport. No aplica porque el formulario siempre está visible (está above the fold).

**Si F0 establece que `client:idle` causa un flash visible** (el formulario se ve y de repente se vuelve interactivo con un salto de layout): cambiar a `client:load`. El Implementador revisa esto y documenta la decisión final en `F0-hallazgos.md` o en el contexto del Implementador de `.agents/`.

**Criterio de aceptación**: el formulario se vuelve interactivo en < 500ms después del paint. No hay flash de contenido no estilizado (FOUC). Los campos responden al click correctamente.

## Tipos exportados por `HomeForm` (para F8)

`HomeForm.tsx` debe exportar los tipos `FormState`, `StyleOption`, `SkinColor` y `BuildValue` para que F8 (Zustand store) los importe directamente sin redefinirlos:

export type { FormState, StyleOption, SkinColor, BuildValue } from './HomeForm'

Esto evita que F8 redefina los mismos tipos y garantiza consistencia entre el estado local de F7 y el store de F8.

## Riesgos

| Riesgo | Mitigación |
|---|---|
| `Badge` sin directiva `client:` no renderiza en SSR si Astro requiere hydration explícita | Convertirlo a componente Astro nativo (`.astro`) si es necesario; la lógica de Badge es solo CSS y children, sin estado |
| `client:idle` causa un flash de ~200ms donde el formulario se ve pero no responde | Cambiar a `client:load` si es perceptible. Documentar la decisión en `F0-hallazgos.md` |
| Los puntos decorativos del "terminal" se perciben como botones interactivos | Agregar `aria-hidden="true"` y `pointer-events: none` en CSS para que no sean accesibles ni clickeables |
| El `max-width: 640px` del contenedor home corta el formulario en xl (600px) con padding | Ajustar a `max-width: min(640px, 100% - 2rem)` para que nunca supere el ancho del viewport menos los márgenes |
| Los tipos `StyleOption`, `SkinColor`, `BuildValue` necesitan estar disponibles en F8 antes de que F7 los exporte | Verificar que el barrel de `HomeForm` los exporta correctamente antes de cerrar F7 |
