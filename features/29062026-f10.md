## Política de errores en el servidor

Esta política aplica a todo código server-side de Modus Ratio — F10 la establece, F11-F14 la heredan.

### Regla: las API routes nunca usan `throw`

Cada handler `export const GET/POST/PUT: APIRoute` retorna una `Response` en **todos** los caminos posibles, incluyendo errores. No se propaga ninguna excepción al runtime de Astro.

```ts
// MAL — si parseBody lanza, Astro recibe una excepción no manejada
export const POST: APIRoute = async ({ request }) => {
  const body = await request.json()   // puede lanzar
  if (!body.file) throw new Error('No file')  // nunca
  return ok({ ok: true })
}

// BIEN — todo camino retorna Response
export const POST: APIRoute = async ({ request }) => {
  let body: unknown
  try {
    body = await request.json()
  } catch {
    return err('INVALID_JSON', 'Request body is not valid JSON', 400)
  }
  if (!isValidBody(body)) return err('BAD_REQUEST', 'Missing required fields', 400)
  return ok({ ok: true })
}
```

**Excepción permitida**: `env.ts` usa `throw` en la función `required()` que se ejecuta en tiempo de carga del módulo (boot), no durante una petición. Crashear en boot con un mensaje claro es la conducta correcta. Esta excepción no aplica a handlers ni a ninguna función llamada desde ellos.

### Regla: las utilidades de infra absorben sus propios errores (fail-open)

Las funciones de `rateLimit.ts` y `redis.ts` capturan internamente sus excepciones y retornan valores seguros. Las API routes y el middleware no necesitan envolver llamadas a estas utilidades en try/catch propios.

**Por qué fail-open y no fail-closed**: si Redis está caído, bloquear a TODOS los usuarios es peor que permitir acceso sin rate limiting por un período corto. El localStorage de F8 sigue operando como capa UX. El fail-open es la decisión correcta para un sistema de protección auxiliar, no para autenticación.

**Cuándo el fallo se registra y cuándo se silencia**:
- `console.error('[rateLimit] ...')` en todos los errores de Redis — siempre visible en logs de Vercel.
- Nunca exponer el mensaje de error de Redis al cliente — solo el código de respuesta estándar.

### T10.1 — `src/lib/env.ts` — Typed env validator

Centralizar y validar todas las variables de entorno del servidor. Es el único archivo que toca `import.meta.env` directamente. Todos los demás módulos importan desde aquí.

**Regla**: si una variable requerida está ausente, lanzar un error explícito al arrancar el servidor — mejor un crash en boot que un error oscuro en runtime durante una petición real.

**Estructura**:

```ts
function required(key: string): string {
  const value = import.meta.env[key]
  if (!value) throw new Error(`[env] Missing required variable: ${key}`)
  return value
}

function optional(key: string, fallback = ''): string {
  return import.meta.env[key] ?? fallback
}

export const env = {
  redis: {
    url: required('REDIS_URL'),
  },
  r2: {
    accountId:       required('R2_ACCOUNT_ID'),
    accessKeyId:     required('R2_ACCESS_KEY_ID'),
    secretAccessKey: required('R2_SECRET_ACCESS_KEY'),
    bucketName:      required('R2_BUCKET_NAME'),
    publicUrl:       required('R2_PUBLIC_URL'),  // URL pública del bucket para lectura
  },
  openai: {
    apiKey: required('OPENAI_API_KEY'),
  },
  minimax: {
    apiKey:  required('MINIMAX_API_KEY'),
    groupId: optional('MINIMAX_GROUP_ID'),
  },
  resend: {
    apiKey: required('RESEND_API_KEY'),
  },
  app: {
    env: optional('NODE_ENV', 'development'),
  },
} as const
```

**Por qué `as const`**: evita que alguien mute `env.redis.url` accidentalmente. Todos los valores son readonly en tiempo de compilación.

**Por qué `import.meta.env` y no `process.env`**: Astro usa `import.meta.env` para las variables de entorno en SSR. `process.env` funciona en algunas configuraciones pero no está garantizado con el adapter de Vercel.

**Criterio de aceptación**: importar `env` desde `@/lib` y acceder a `env.redis.url` en un archivo server-side compila sin errores. Si se elimina `REDIS_URL` del `.env`, el servidor lanza el error de boot al instanciar el módulo.

### T10.2 — `src/lib/redis.ts` — Cliente Redis singleton

**Verificación previa**: confirmar si `ioredis` está instalado desde F1. Si no: `pnpm add ioredis`. Verificar también la versión de ioredis que usa AzanoRivers/Optimus para usar la misma.

**Singleton pattern**: en Astro SSR con Vercel, cada función serverless puede recibir múltiples requests en la misma instancia. El singleton evita crear una conexión por request.

```ts
import { Redis } from 'ioredis'
import { env } from '@/lib/env'

let _client: Redis | null = null

function getClient(): Redis {
  if (!_client) {
    _client = new Redis(env.redis.url, {
      lazyConnect: true,
      maxRetriesPerRequest: 3,
      enableReadyCheck: false,
      keyPrefix: '',  // NO usar keyPrefix de ioredis — los prefijos se gestionan manualmente en rateLimit.ts
    })

    _client.on('error', (err) => {
      console.error('[redis] connection error:', err.message)
    })
  }
  return _client
}

export const redis = getClient()
```

**Por qué `keyPrefix: ''` en lugar de `'modusratio:'`**: el `keyPrefix` de ioredis aplica a TODAS las operaciones, incluyendo las de pipelines y scripts Lua. Es más seguro aplicar el prefijo manualmente en `rateLimit.ts` para tener control explícito. Si AzanoRivers usa el mismo cliente y un `keyPrefix` global diferente, el de MR colisionaría.

**Por qué `lazyConnect: true`**: no conecta hasta el primer comando. Evita errores de boot si Redis no está disponible al iniciar el servidor.

**Por qué `enableReadyCheck: false`**: en entornos serverless, los sockets pueden estar en cualquier estado. `enableReadyCheck: false` hace que ioredis comience a aceptar comandos inmediatamente sin esperar el handshake de `READY`.

**Criterio de aceptación**: `redis.ping()` retorna `'PONG'` en un endpoint de diagnóstico. Si Redis no está disponible, el error es visible en logs sin crashear la app silenciosamente.

### T10.3 — `src/lib/getIP.ts` — Extracción de IP real

Detrás de Vercel, la IP del cliente llega en el header `x-forwarded-for` como una lista separada por comas: `"123.45.67.89, 10.0.0.1, 172.16.0.2"`. El primer valor es la IP del cliente real.

```ts
export function getIP(request: Request): string {
  const forwarded = request.headers.get('x-forwarded-for')
  if (forwarded) {
    return forwarded.split(',')[0].trim()
  }
  // Fallback para desarrollo local sin proxy
  return '127.0.0.1'
}
```

**Por qué tomar el primer valor y no el último**: Vercel agrega su propia IP al final de la cadena. El cliente real es siempre el primero. Tomar el último daría la IP del edge de Vercel, no la del usuario.

**Sobre spoofing de `x-forwarded-for`**: un cliente malicioso puede enviar el header con una IP falsa. Vercel antepone la IP real del socket al frente de la cadena, pero si el cliente envía `x-forwarded-for: 1.1.1.1` y Vercel añade la IP real, el resultado es `1.1.1.1, <ip_real>`. Tomar el primer valor sería la IP spoofada. Mitigación: documentar que la IP del header no es 100% confiable contra ataques sofisticados, pero es suficiente para el uso legítimo y el rate limiting casual. Para producción, si Vercel Firewall está activo, usa el header `x-vercel-forwarded-for` que es más difícil de falsificar.

**Criterio de aceptación**: en dev local, `getIP(request)` retorna `'127.0.0.1'`. En producción (Vercel), retorna la IP real del cliente.

### T10.4 — `src/lib/rateLimit.ts` — Helpers de ventana deslizante y ban

Este archivo contiene toda la lógica de rate limit del servidor. Es el espejo Redis del sistema localStorage de F8.

**Keys de Redis**:

```ts
const USAGE_KEY = (ip: string) => `modusratio:usage:${ip}`
const BAN_KEY   = (ip: string) => `modusratio:ip:${ip}:banned`

const USAGE_LIMIT      = 25
const USAGE_WINDOW_MS  = 60 * 60 * 1000  // 1 hora en ms
const USAGE_WINDOW_SEC = 60 * 60          // 1 hora en segundos (para TTL de Redis)
const BAN_TTL_SEC      = 60 * 60 * 24     // 24 horas de ban
const BAN_MULTIPLIER   = 3                // USAGE_LIMIT * 3 = 75 → ban automático
```

**Interfaz de retorno de `checkUsage`**:

```ts
export interface UsageStatus {
  allowed: boolean
  remaining: number          // análisis restantes en la ventana actual
  resetAt: number | null     // timestamp ms en que el más antiguo expira, null si no hay registros
  shouldBan: boolean         // true si el conteo supera BAN_MULTIPLIER * USAGE_LIMIT
}
```

**Valor de retorno cuando Redis falla** (fail-open):

```ts
const FAIL_OPEN_STATUS: UsageStatus = {
  allowed:   true,
  remaining: USAGE_LIMIT,
  resetAt:   null,
  shouldBan: false,
}
```

**Implementación de cada función con manejo de error interno**:

Las cuatro funciones públicas capturan sus propios errores. Ninguna propaga excepciones al caller. El Implementador aplica este patrón a cada una:

`checkUsage(ip)`:
1. Envolver todo en try/catch.
2. Dentro del try: `ZREMRANGEBYSCORE` → `ZRANGE 0 0 WITHSCORES` → `ZCARD` → calcular y retornar `UsageStatus`.
3. En el catch: `console.error('[rateLimit] checkUsage failed:', e)` → retornar `FAIL_OPEN_STATUS`.

`recordUsage(ip)`:
1. Envolver en try/catch.
2. Dentro del try: `ZADD <now> <now>` → `EXPIRE <USAGE_WINDOW_SEC>`.
3. En el catch: `console.error('[rateLimit] recordUsage failed:', e)` → no-op (la función retorna void).

`banIP(ip)`:
1. Envolver en try/catch.
2. Dentro del try: `SET <BAN_KEY> "1" EX <BAN_TTL_SEC>`.
3. En el catch: `console.error('[rateLimit] banIP failed:', e)` → no-op.

`isBanned(ip)`:
1. Envolver en try/catch.
2. Dentro del try: `EXISTS <BAN_KEY>` → retornar `result === 1`.
3. En el catch: `console.error('[rateLimit] isBanned failed:', e)` → retornar `false` (fail-open: no bloquear a nadie por un error de Redis).

**Uso en API routes** — las utilidades nunca lanzan, el route nunca necesita try/catch para ellas:

```ts
// Dentro del endpoint POST /api/analyze (F11)
const ip = getIP(context.request)
const status = await checkUsage(ip)   // nunca lanza

if (!status.allowed) {
  if (status.shouldBan) await banIP(ip)   // nunca lanza
  return err('USAGE_LIMIT_EXCEEDED', 'Hourly limit reached', 429)
}

await recordUsage(ip)   // nunca lanza
if (status.shouldBan) await banIP(ip)
// ... continuar
```

**Por qué `shouldBan` se chequea tanto antes como después del record**: si el count ya estaba en el umbral antes de registrar, el ban se activa en el check. Si el record lleva el count al umbral, se activa después. El ban es idempotente (`SET ... EX` sobrescribe si ya existe).

**Sobre las operaciones Redis independientes vs pipeline**: `ZREMRANGEBYSCORE`, `ZRANGE` y `ZCARD` pueden agruparse en un pipeline de ioredis para reducir round-trips. En la primera iteración se hacen secuenciales para claridad y se optimizan en F17 si hay latencia perceptible.

**Criterio de aceptación**:
- Con Redis disponible: `recordUsage('test-ip')` ×25, luego `checkUsage('test-ip')` → `{ allowed: false, remaining: 0 }`.
- Con Redis caído (simular desconectando): `checkUsage('any-ip')` retorna `FAIL_OPEN_STATUS` sin lanzar. El endpoint que la llama sigue retornando una `Response` válida.

### T10.5 — `src/lib/apiResponse.ts` — Helpers de respuesta JSON

Convención estándar para todos los API routes de la app:

```ts
export function ok<T>(data: T, status = 200): Response {
  return Response.json({ data }, { status })
}

export function err(code: string, message: string, status: number): Response {
  return Response.json({ error: { code, message } }, { status })
}
```

**Códigos de error estandarizados** (documentados aquí, usados en F11-F14):

| `code` | `status` | Situación |
|---|---|---|
| `USAGE_LIMIT_EXCEEDED` | 429 | El IP superó las 25 imágenes/hora |
| `FILE_TOO_LARGE` | 413 | La imagen supera el tamaño máximo |
| `INVALID_MIME_TYPE` | 415 | El tipo de archivo no está permitido |
| `R2_UPLOAD_FAILED` | 502 | Error al generar la URL presignada de R2 |
| `AI_ANALYSIS_FAILED` | 502 | Error en la llamada a GPT-4o-mini o Minimax |
| `INTERNAL_ERROR` | 500 | Error no esperado |

**Por qué `code` es un string y no un número**: los clientes React pueden hacer `if (error.code === 'USAGE_LIMIT_EXCEEDED')` sin depender de números mágicos de HTTP. El status HTTP es para el protocolo; el code es para la lógica de la UI.

**Regla de uso en API routes**: cada handler retorna siempre una `Response`. Si una operación puede fallar, se envuelve en try/catch local que retorna `err(...)`. Nunca se deja propagar una excepción al runtime de Astro. Ver la "Política de errores en el servidor" al inicio de este documento.

Patrón de un route completo con operaciones que pueden fallar:

```ts
export const POST: APIRoute = async ({ request, locals }) => {
  // Parsear body — puede fallar si el cliente envía JSON inválido
  let body: unknown
  try {
    body = await request.json()
  } catch {
    return err('INVALID_JSON', 'Request body is not valid JSON', 400)
  }

  // Validar campos — lógica pura, no lanza
  if (!isValidBody(body)) return err('BAD_REQUEST', 'Missing required fields', 400)

  // Llamadas a utilidades de infra — ya capturan sus propios errores, no necesitan try/catch
  const ip = getIP(request)
  const status = await checkUsage(ip)
  if (!status.allowed) return err('USAGE_LIMIT_EXCEEDED', 'Hourly limit reached', 429)

  // Llamada a servicio externo — puede fallar, envolver
  let result: ExternalResult
  try {
    result = await callExternalService(body)
  } catch {
    return err('EXTERNAL_SERVICE_FAILED', 'External service error', 502)
  }

  return ok(result)
}
```

**Criterio de aceptación**: un API route que retorna `ok({ hello: 'world' })` produce `{"data":{"hello":"world"}}` con status 200. Un route que retorna `err('INTERNAL_ERROR', 'Unexpected', 500)` produce `{"error":{"code":"INTERNAL_ERROR","message":"Unexpected"}}` con status 500.

### T10.8 — `src/middleware.ts` — Middleware Astro

Archivo nuevo en `src/middleware.ts` (mismo nivel que `src/pages/`).

**Responsabilidades**:
1. Inyectar `context.locals.language` para que todas las páginas Astro usen `Astro.locals.language` en lugar de llamar `getLocale(Astro.request)` directamente.
2. Verificar ban de Redis y redirigir a `/429` si corresponde — excepto para la ruta `/429` misma (evita loop).

**Rutas excluidas del ban check**:
- `/429` — la página de error misma
- `/_astro/*` — assets estáticos de Astro
- `/favicon.*` — favicon

```ts
import { defineMiddleware } from 'astro:middleware'
import { getLocale } from '@/i18n'
import { getIP } from '@/lib/getIP'
import { isBanned } from '@/lib/rateLimit'

const BAN_CHECK_EXCLUDED = ['/429', '/_astro', '/favicon']

function isExcluded(pathname: string): boolean {
  return BAN_CHECK_EXCLUDED.some(prefix => pathname.startsWith(prefix))
}

export const onRequest = defineMiddleware(async (context, next) => {
  // 1. Inyectar locale en locals
  context.locals.language = getLocale(context.request)

  // 2. Ban check
  if (!isExcluded(context.url.pathname)) {
    const ip = getIP(context.request)
    if (await isBanned(ip)) {
      return context.redirect('/429', 302)
    }
  }

  return next()
})
```

**Por qué el locale se inyecta incluso en rutas de API**: las API routes también pueden necesitar `locals.language` para construir mensajes de error localizados si en el futuro se decide enviar mensajes en el idioma del usuario.

**Por qué el middleware no necesita su propio try/catch**: `isBanned` captura sus errores internamente (T10.4) y retorna `false` si Redis falla. `getLocale` y `getIP` son funciones puras que no tocan I/O. El resultado es que `onRequest` no tiene rutas de excepción. Si en algún caso inesperado el middleware lanzara una excepción, Astro la capturaría y serviría el `500.astro` — comportamiento aceptable y consistente con la política de errores.

**Criterio de aceptación**: en cualquier `.astro` file, `Astro.locals.language` tiene valor `'es'` o `'en'`. Simular un ban en Redis y acceder a `/` redirige a `/429`. Acceder a `/429` directamente NO redirige (no hay loop). Con Redis caído: el middleware no bloquea ninguna petición — todas pasan a `next()` con fail-open.

### T10.11 — Endpoint de diagnóstico (solo en `DEV`)

Crear `src/pages/api/health.ts` — endpoint de verificación que solo responde en desarrollo:

```ts
import type { APIRoute } from 'astro'
import { redis } from '@/lib/redis'
import { ok, err } from '@/lib/apiResponse'

export const GET: APIRoute = async () => {
  if (import.meta.env.PROD) {
    return err('NOT_FOUND', 'Not found', 404)
  }

  try {
    const pong = await redis.ping()
    return ok({ redis: pong, status: 'ok' })
  } catch (e) {
    return err('REDIS_UNAVAILABLE', String(e), 503)
  }
}
```

**Por qué solo en DEV**: el endpoint expone información sobre el estado interno del sistema. En producción debe ser inaccesible.

**Por qué este endpoint sí tiene try/catch cuando la política dice no-throw**: el try/catch aquí no es para manejar un error de lógica de negocio, sino para testear intencionalmente la conectividad de Redis y reportar su estado al desarrollador. Si Redis lanza, el handler retorna `err('REDIS_UNAVAILABLE', ...)` — sigue siendo una `Response` válida, no una excepción propagada. Es el único caso en el codebase donde un try/catch rodea una operación Redis directamente en un handler, y está justificado porque es un endpoint de diagnóstico, no de negocio.

**Criterio de aceptación**: `GET /api/health` en dev con Redis operativo retorna `{"data":{"redis":"PONG","status":"ok"}}`. Con Redis caído retorna `{"error":{"code":"REDIS_UNAVAILABLE",...}}` con status 503. En producción retorna 404 sin consultar Redis.

## Riesgos

| Riesgo | Mitigación |
|---|---|
| `ioredis` en entorno serverless Vercel puede agotar conexiones si no hay pool | Usar singleton con `lazyConnect`. Si el problema persiste en F19 (deploy), evaluar migrar a `@upstash/redis` que es HTTP-based y sin estado de conexión |
| La IP `x-forwarded-for` puede ser spoofada | Documentado en T10.3. Si Vercel Firewall está activo, usar `x-vercel-forwarded-for` que es más confiable. Ajuste en F19 si se activa el firewall |
| El middleware agrega latencia Redis en CADA request (incluyendo páginas estáticas) | El `isBanned` es un `EXISTS` de Redis — sub-ms. Si hay páginas con muchos assets, excluirlas en `BAN_CHECK_EXCLUDED`. Medir en F19 |
| `import.meta.env` puede no estar disponible en algunos contextos de middleware | Astro garantiza `import.meta.env` en middleware SSR. Si hay problemas, usar `process.env` como fallback en `env.ts` |
| `500.astro`: `Astro.locals` puede no estar seteado si el middleware falló antes de llegar a `next()` | El try/catch de F9 para locale se mantiene como safety net. No se elimina aunque el middleware exista |
| Sorted set de Redis con timestamps repetidos (dos requests exactamente en el mismo ms) | `ZADD` con score = member = timestamp: si score y member coinciden, se sobreescribe en lugar de duplicar. Mitigación: usar `uuid` como member y timestamp como score. En F10 se acepta timestamp como member (riesgo muy bajo en práctica) |
