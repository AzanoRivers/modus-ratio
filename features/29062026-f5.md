---

## Objetivo

Producir dos mecanismos de transición complementarios:

1. **Transición de página** (entrada al cargar cada página Astro): efecto CRT de barrido vertical con scanlines. Se activa una vez al montar la página. Usa solo CSS.
2. **Transición de estado** (entre vistas dentro de la misma página): componente React `CyberTransition` que dispara el efecto cuando cambia su `transitionKey`. Se usa para los cambios `formulario → loader → resultados → formulario`.

Ambos mecanismos respetan `prefers-reduced-motion` y tienen una versión aligerada para móvil.

---

## Contexto: por qué dos mecanismos distintos

En `azanolabs-web` (Next.js, SPA), `PageTransitionWrapper` usa `usePathname()` + `key={pathname}` para disparar la animación en cada cambio de ruta. Ese patrón no aplica a Modus Ratio porque:

- Modus Ratio tiene **una sola URL** (`/`). No hay navegación entre rutas durante el flujo principal.
- Los cambios que necesitan transición son cambios de **estado** (`idle → uploading → analyzing → results`), no de ruta.
- Las páginas de error (`/404`, `/500`, `/429`) sí son rutas distintas, pero se llega a ellas por redirección del servidor o error, no por navegación del usuario desde el flujo principal.

**Conclusión**:
- `CyberTransition` (state-driven, basado en `key`) es el mecanismo principal.
- La transición de página (CSS puro) cubre la entrada inicial a cualquier ruta.

**Contenido a agregar** (port directo de la referencia):

```css
/* ── Transición de entrada de página — CRT cyberpunk ─────────────────────── */

.page-enter {
  animation: cyberCRTEnter 380ms cubic-bezier(0, 0, 0.2, 1) both;
  position: relative;
  isolation: isolate;
  will-change: clip-path, filter;

.page-enter::after {
  content: "";

@keyframes cyberCRTEnter {

  20%  { clip-path: inset(65dvh 0 0 0);  filter: brightness(3) saturate(0.1) hue-rotate(40deg); }

  75%  { clip-path: inset(0 0 0 0);      filter: brightness(1.1) saturate(0.95); }
  88%  { filter: brightness(1.05) saturate(1.01); }
  100% { clip-path: inset(0 0 0 0);      filter: brightness(1) saturate(1) hue-rotate(0deg); }
}

@keyframes scanlinesFade {
  0%   { opacity: 1; }
  70%  { opacity: 0.7; }
  100% { opacity: 0; }
}

  to   { background-position: 0 4px; }
}

/* Reduce-motion global — desactiva TODAS las animaciones de transición */
@media (prefers-reduced-motion: reduce) {
  .page-enter,
  .page-enter::after {
    animation: none;
  }
}

/* Versión móvil: sin filter (costoso en GPU de gama baja), menos keyframes */
/* En Modus Ratio xl = 600px, por tanto móvil = < 600px                     */
@media (max-width: 599px) {
  .page-enter {
    animation: cyberCRTEnterMobile 280ms cubic-bezier(0, 0, 0.2, 1) both;
    will-change: clip-path, opacity;
  }
  .page-enter::after {
    animation: scanlinesFade 320ms ease-out both;
  }
}

@keyframes cyberCRTEnterMobile {
  0%   { clip-path: inset(100dvh 0 0 0); opacity: 0.6; }
```

**Ajuste respecto a la referencia**: el media query de la versión móvil cambia de `max-width: 1279px` (que en azanolabs-web separaba móvil de desktop en su `xl:1280`) a `max-width: 599px` (que en Modus Ratio separa móvil de desktop en su `xl:600`). El efecto es el mismo: desktop recibe el CRT completo con `filter`, móvil recibe la versión ligera sin `filter`.

**Nota sobre `-webkit-clip-path`**: la referencia incluye ambos prefijos (`-webkit-clip-path` y `clip-path`). En la mayoría de navegadores modernos (2026) `clip-path` sin prefijo es suficiente. Sin embargo, dado que el proyecto debe funcionar en Safari/iPhone (F18), se mantienen ambos prefijos.

**Criterio de aceptación**: al cargar la página demo (T5.5), el efecto CRT de barrido vertical es visible en desktop. En móvil (emulado en DevTools), la versión ligera (sin `filter`, sin `scanlinesRoll`) es la que se dispara.

### T5.2 — Mecanismo de transición de página en Astro

El CSS del T5.1 define la animación, pero necesita que el elemento tenga la clase `.page-enter` al montarse en el DOM. Este task define cómo aplicar esa clase en Astro.

**Opción elegida: clase aplicada en `BaseLayout.astro`**

El elemento `<main>` del `BaseLayout.astro` recibe la clase `page-enter` directamente:

```astro
<main class="page-enter flex-1 flex flex-col">
  <slot />
</main>
```

**Por qué funciona**: en Astro SSR, cada navegación es una nueva request → nueva respuesta HTML → el browser monta el DOM desde cero → el elemento `<main class="page-enter">` es nuevo → el browser dispara la animación CSS automáticamente. Sin JavaScript de cliente.

**Relación con Astro View Transitions**: F0 (T0.6) investiga si las View Transitions nativas de Astro son compatibles con las transiciones VHS custom. Si F0 concluye que las View Transitions interfieren (p. ej. double-animation al navegar entre páginas), se desactivan con `<ViewTransitions />` omitido o con `transition:persist` en el `<main>`. Si son compatibles y suman, se puede agregar `<ViewTransitions />` en el `<head>` del layout para que las navegaciones sean SPA-like y el CRT se dispare como transición nativa de Astro.

**Decisión a tomar según F0**:
- Si F0 dice: "View Transitions + CSS custom son compatibles" → agregar `<ViewTransitions />` al layout e integrar la animación con la API de View Transitions.
- Si F0 dice: "colisionan" → quedarse con la clase `page-enter` directa en `<main>` (approach actual) y omitir View Transitions nativas.
- El sub-plan de F5 documenta ambas opciones; el Implementador elige según `F0-hallazgos.md`.

**Criterio de aceptación**: navegar entre dos páginas Astro (home → 404, o cargando directamente `/404`) produce el efecto CRT. No aparece doble animación ni flash en blanco.

### T5.3 — Componente `CyberTransition` (React, state-driven)

Port de `CyberTransition.tsx` de azanolabs-web adaptado a las convenciones de Modus Ratio (sin CSS modules, sin `next/navigation`, con el `.css` del componente).

**Ubicación**: `src/components/atoms/CyberTransition/`

**Props**:

```ts
interface CyberTransitionProps {
  transitionKey: string | number   // al cambiar este valor, el componente se remonta y dispara la animación
  effect?: 'scanIn' | 'slideUp' | 'glitch'   // default: 'scanIn'
  className?: string
  children: React.ReactNode
}
```

**Implementación** (sin cambios de lógica respecto a la referencia):

```tsx
export function CyberTransition({
  transitionKey,
  children,
  effect = 'scanIn',
  className,
}: CyberTransitionProps) {
  return (
    <div
      key={transitionKey}
      data-effect={effect}
      className={['cyber-transition', className].filter(Boolean).join(' ')}
    >
      {children}
    </div>
  )
}
```

**Diferencias respecto a azanolabs-web**:
- No usa `styles.wrapper` (CSS module): usa la clase global `cyber-transition` definida en `CyberTransition.css`.
- No importa `styles` de un `.module.css`: importa el `.css` del componente.
- No tiene `"use client"`: en Astro, el componente se hidrata con la directiva del padre (`client:load` / `client:idle`). El `"use client"` solo existe en Next.js.

**Criterio de aceptación**: el componente exporta `CyberTransition` correctamente desde el barrel `index.ts`. TypeScript acepta los 3 valores de `effect`. Cambiar `transitionKey` en la demo produce la animación.

### T5.4 — CSS de `CyberTransition`

Crear `src/components/atoms/CyberTransition/CyberTransition.css` con las 3 animaciones y el soporte de `prefers-reduced-motion`.

**Diferencia con la referencia**: en azanolabs-web era un CSS module (`.module.css`) con clases locales (`.wrapper[data-effect="..."]`). En Modus Ratio es un CSS global del componente con el selector de clase global:

```css
.cyber-transition[data-effect="scanIn"] {
  animation: ctScanIn 180ms ease-out both;
}

.cyber-transition[data-effect="slideUp"] {
  animation: ctSlideUp 200ms ease-out both;
}

.cyber-transition[data-effect="glitch"] {
  animation: ctGlitchIn 100ms steps(1) both;
}

@keyframes ctScanIn {
  from {
    -webkit-clip-path: inset(0 0 100% 0);
    clip-path: inset(0 0 100% 0);
    opacity: 0.4;
  }
  to {
    -webkit-clip-path: inset(0 0 0% 0);
    clip-path: inset(0 0 0% 0);
    opacity: 1;
  }
}

@keyframes ctSlideUp {
  from { transform: translateY(0.5rem); opacity: 0; }
  to   { transform: translateY(0);      opacity: 1; }
}

@keyframes ctGlitchIn {
  0%   { transform: translateX(-0.25rem);  opacity: 0; }
  50%  { transform: translateX(0.1875rem); opacity: 0.8; }
  100% { transform: translateX(0);         opacity: 1; }
}

@media (prefers-reduced-motion: reduce) {
  .cyber-transition[data-effect] {
    animation: none;
  }
}
```

**Por qué se prefijan los `@keyframes` con `ct`**: para evitar colisión de nombres con los keyframes globales de `globals.css` (`cyberCRTEnter`, `scanlinesFade`, etc.). Los nombres globales no tienen prefijo; los del componente tienen `ct` (CyberTransition).

**Cómo se importa el CSS**: en Astro, cada `.astro` que use `CyberTransition` puede importar el `.css` directamente, o el componente puede tener el `<style>` inline en su wrapper `.astro`. Sin embargo, para seguir la convención de diseño atómico, el Implementador debe verificar en `F0-hallazgos.md` (T0.4) si Astro carga automáticamente el `.css` de un componente `.tsx` cuando está en la misma carpeta, o si hay que importarlo explícitamente en el `.astro` que lo usa.

**Criterio de aceptación**: los 3 efectos producen su animación correspondiente en la demo (T5.5). `prefers-reduced-motion` desactiva los 3. Ningún valor de color ni medida aparece hardcodeado fuera de las variables de `globals.css`.

## Contrato de uso de `CyberTransition` (para F6 en adelante)

Este contrato define cómo todos los componentes de las fases siguientes usan `CyberTransition`.

**Cuándo usar `CyberTransition`**:
- Cualquier cambio de vista dentro de la misma página: formulario ↔ loader ↔ resultados.
- Cualquier componente que "reemplaza" a otro en el mismo contenedor.
- Cualquier expansión/colapso de sección que no sea solo `height: auto → 0`.

**Cuándo NO usar `CyberTransition`**:
- Hover states (usar CSS `:hover` con `transition-property`).
- Cambios de estado booleano dentro de un mismo componente que no reemplazan el bloque completo (p. ej. cambiar el texto de un botón al cargar).
- Scrolling (ninguna animación en scroll; el usuario controla el scroll).

**Regla de `transitionKey`**:
- El `transitionKey` debe cambiar SOLO cuando el bloque de contenido cambia completamente.
- No debe ser un timestamp o un número aleatorio: debe ser un valor semántico (el nombre del phase, el id del resultado, etc.).
- Ejemplo correcto: `transitionKey={phase}` donde `phase` es `'idle' | 'analyzing' | 'results'`.
- Ejemplo incorrecto: `transitionKey={Date.now()}`.

**Efecto por defecto recomendado**:
- `scanIn`: para cambios de fase principales (el más vistoso, debe usarse con moderación).
- `slideUp`: para expansión de secciones secundarias o aparición de badges de advertencia.
- `glitch`: para estados de error o advertencia (efecto de fallo de señal).

**Duración**:
- `scanIn`: 180ms. No superar este valor para mantener la fluidez.
- `slideUp`: 200ms. Apropiado para contenido secundario.
- `glitch`: 100ms. Instantáneo, para mensajes de error.
- Todas las duraciones son fijas en el CSS. No se pasan por props ni por `style={{}}`.

## Riesgos

| Riesgo | Mitigación |
|---|---|
| Astro View Transitions interfiere con la clase `page-enter` causando doble animación | Confirmar en F0 (T0.6). Solución de fallback: omitir `<ViewTransitions />` y quedarse con la clase `page-enter` directa en `<main>` |
| Safari / iPhone no soporta `dvh` en los keyframes | F18 cubre la revisión completa de bugs de iPhone. En F5 documentar la dependencia: si `dvh` falla en Safari, reemplazar por `100vh` con un wrapper `min-height: 100svh` |
| El CSS de `CyberTransition` no se carga automáticamente en Astro al importar el `.tsx` | Confirmar en F0 (T0.4). Solución: importar el `.css` explícitamente en el `.astro` que usa el componente: `import '@/components/atoms/CyberTransition/CyberTransition.css'` |
| `key={transitionKey}` causa que React desmonte y remonte el árbol completo incluyendo estados internos de hijos | El contrato de uso (T5.5) especifica que `CyberTransition` solo envuelve bloques que se reemplazan completamente; no debe envolver stores o contextos persistentes |
| La versión móvil sin `filter` pierde parte del efecto neón | Es intencional: el `filter` en keyframes es costoso en GPU de gama baja. El clip-path sigue dando el efecto de barrido; la pérdida de color es aceptable en móvil |
