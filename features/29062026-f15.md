# 29062026-f15 — Loader energético

> Sub-plan de la **Fase 15** del fundational plan (`29062026-fundation-plan.md`).
> Componente `AnalysisLoader` que ocupa la vista durante las fases `uploading` y `analyzing`. Muestra un orbe de energía con rayos de color y un temblor que se intensifica a medida que aumenta el progreso, usando CSS custom property `--progress` alimentada desde el store.

- **Fase**: F15 - Loader energético

- **Estado**: cerrada (implementada 01-07-2026)
- **Prerequisitos**: F5 (`CyberTransition`, keyframes `ct`-prefijados), F8 (store con `phase`, `uploadProgress`)

## Bug post-cierre (01-07-2026): el loader nunca se veía

Al conectarlo en F17, `AnalysisLoader` nunca aparecía en pantalla durante `uploading`/`analyzing` (el formulario desaparecía y no se veía nada hasta los resultados). Diagnosticado con logs explícitos en `FlowController`/`AnalysisLoader`/`useAnalysisFlow`: el componente **sí se renderizaba** correctamente con los props correctos, así que no era un bug de lógica/React.

**Causa real**: `CyberTransition` (el padre directo de `AnalysisLoader` en el árbol) anima `clip-path` para el efecto de entrada VHS (`animation: ctScanIn ... both`). El `both` fill-mode deja el `clip-path` final (`inset(0 0 0% 0)`, visualmente un no-op) aplicado de forma **permanente** al div de `CyberTransition` una vez termina la animación. Un `clip-path` distinto de `none` convierte a ese elemento en el contenedor de referencia de sus descendientes `position: fixed` (en vez del viewport), y como ese div no tiene contenido que le dé tamaño propio (su único hijo es el propio loader, que al ser `position:fixed` no aporta a su flujo), su caja colapsaba a 0 de alto, encogiendo al loader ahí adentro.

Un intento anterior (quitar `filter` del `will-change` de `.page-enter`) no lo resolvió porque la causa real estaba un nivel más abajo, en `CyberTransition`, no en `.page-enter`.

**Fix**: `AnalysisLoader` ahora se renderiza con `createPortal(loader, document.body)` de `react-dom`, sacándolo por completo de la jerarquía de `CyberTransition`/`.page-enter`. Queda garantizado que se posiciona contra el viewport real sin depender de qué CSS tengan sus ancestros. Si se agrega otro overlay de pantalla completa en el futuro, aplicar el mismo patrón (portal a `document.body`) en vez de confiar en `position: fixed` dentro del árbol normal.

## Objetivo

Cuando el usuario sube una imagen y espera el análisis de IA (~8-10s en total), la vista debe comunicar que algo energético está pasando. No es un spinner neutral: es un efecto visual de carga que crece en intensidad conforme avanza el proceso, culminando en un temblor máximo durante el análisis de IA.

El componente aparece en dos fases del store:
- `uploading`: progreso real (0→100) desde `uploadProgress`, temblor creciente
- `analyzing`: progreso fijo a `1` (máxima intensidad), temblor continuo de alta frecuencia

## Cierre — Definition of Done

- Cuando `phase === 'uploading'`: el loader aparece, el orbe pulsa suavemente con `--progress = uploadProgress / 100`. El texto muestra `t.loader.uploading`.
- Cuando `uploadProgress >= 90`: el texto cambia a `t.loader.almostDone`.
- Cuando `phase === 'analyzing'`: `--progress = 1`, el temblor es máximo y continuo. El texto muestra `t.loader.analyzing`.
- Con `prefers-reduced-motion: reduce`: todas las animaciones se desactivan. El texto y el orbe estático siguen siendo visibles.
- Sin `style={{}}` salvo la CSS custom property `--progress`.
- Ningún texto hardcodeado.
- `pnpm build` sin errores TypeScript.

**Notas de implementación**:

- `calc(0.6s - var(--progress, 0) * 0.45s)`: con `--progress = 0` la duración del shake es `0.6s` (lento, suave). Con `--progress = 1` es `0.15s` (muy rápido, tremor máximo). Esto crea la sensación de energía creciente sin JavaScript adicional.
- `calc(3s - var(--progress, 0) * 1.5s)` en los rayos: la rotación también acelera con el progreso.
- `opacity: calc(0.3 + var(--progress, 0) * 0.5)`: los rayos son más visibles conforme aumenta el progreso.
- `mask-image` en los rayos: crea el efecto de halo separado del orbe central (sin tapar el brillo del orbe).
- Los `var(--color-neon-blue)`, `var(--color-neon-pink)`, `var(--color-neon-green)`, `var(--color-bg)`, `var(--font-mono)` deben existir en `globals.css` (definidos en F4). El Implementador verifica los nombres exactos antes de usar.

**Por qué `calc()` con CSS custom properties en keyframes**: las CSS custom properties son válidas en valores de propiedad (`animation-duration`, `opacity`, `transform`), pero en `@keyframes` el valor se evalúa en tiempo de animación — no en tiempo de cálculo del keyframe. Esto significa que `calc(... var(--progress, 0) ...)` dentro de un `@keyframes` puede no funcionar en todos los navegadores. La forma segura es usarlos en propiedades estáticas como `animation-duration` y `opacity` del elemento, no dentro de los `@keyframes` directamente.

**Revisión necesaria**: el Implementador verifica en F0 si los keyframes con `calc(var(--progress))` en `transform` funcionan en los targets (Safari iOS, Chrome Android). Si no, la alternativa es definir dos animaciones separadas (`ctOrbShakeGentle` / `ctOrbShakeMax`) y alternar entre ellas con una clase CSS (`.analysis-loader--analyzing`), sin custom property dinámica en el shake.

**Criterio de aceptación**: el orbe tiembla más rápido conforme `--progress` se acerca a `1`. Los rayos rotan y son más opacos con progreso alto. Con `prefers-reduced-motion` el orbe y rayos son estáticos.

### T15.3 — `AnalysisLoader.tsx`

Crear `src/components/organisms/AnalysisLoader/AnalysisLoader.tsx`.

```tsx
import './AnalysisLoader.css'
import { useAppStore } from '@/store'
import type { Translations } from '@/i18n/types'

interface AnalysisLoaderProps {
  t: Translations['loader']
}

export function AnalysisLoader({ t }: AnalysisLoaderProps) {
  const phase          = useAppStore((s) => s.phase)
  const uploadProgress = useAppStore((s) => s.uploadProgress)

  const progress =
    phase === 'analyzing'
      ? 1
      : uploadProgress / 100

  const label =
    phase === 'analyzing'
      ? t.analyzing
      : uploadProgress >= 90
        ? t.almostDone
        : t.uploading

  return (
    <div className="analysis-loader">
      <div
        className="analysis-loader__orb"
        style={{ '--progress': progress } as React.CSSProperties}
      />
      <p className="analysis-loader__text">{label}</p>
    </div>
  )
}
```

**Por qué `style={{ '--progress': progress }}`**: esta es la única forma de pasar un valor calculado en tiempo de ejecución a CSS. Es la excepción explícitamente permitida en las reglas del proyecto (CSS custom property con valor dinámico de JS). No hay alternativa en CSS puro para recibir un número 0–1 calculado desde el estado del store.

**Por qué `t` como prop en lugar de leer del store o contexto global**: el diccionario se inyecta desde la página Astro (server-side), que lo recibe como `Astro.locals.language` → `getDictionary(locale)`. Las islas React reciben solo lo que necesitan vía props. Este patrón está establecido desde F3.

**Nota sobre hidratación**: F17 decide la directiva `client:*`. El Orquestador confirma si `client:idle` es suficiente (el loader no necesita estar listo antes de la primera interacción del usuario) o si debe ser `client:load` (para no tener flash de loader sin hidratar). El componente no importa cuál usa — solo lee del store.

**Criterio de aceptación**: TypeScript acepta el tipo de `t`. `--progress` se pasa como CSS custom property. El componente no usa ningún otro `style={{}}`. El texto mostrado corresponde a la fase y progreso actuales.

## Riesgos

| Riesgo | Mitigación |
|---|---|
| `calc(var(--progress))` en `transform` dentro de `@keyframes` no funciona en Safari iOS | Verificar en T15.2. Si falla: usar dos clases CSS (`.analysis-loader--uploading` / `.analysis-loader--analyzing`) y alternar desde el componente según `phase`. El shake de upload usa una animación fija suave; el shake de analyzing usa una animación fija de máxima intensidad |
| Variables CSS de color (`--color-neon-blue`, etc.) tienen nombres distintos en F4 | El Implementador abre `src/styles/globals.css` y verifica los nombres exactos antes de referenciarlos en `AnalysisLoader.css` |
| El loader fijo (`position: fixed`) bloquea interacciones del usuario | Correcto por diseño: durante `uploading` y `analyzing` el usuario no debe interactuar. F17 controla que el loader solo aparezca en esas fases |
| Rendimiento en móvil: `conic-gradient` + rotación puede ser costoso en GPU de gama baja | Si se detecta en QA (F19): simplificar `::before`/`::after` a gradientes radiales simples sin rotación. Priorizar la versión final sobre la animada si hay frame drops |
| `uploadProgress` llega a 100 antes de que el servidor termine de procesar el upload | `uploadProgress` viene del `XHR.onprogress` (bytes transferidos), no de la respuesta del servidor. El 100% significa que los bytes llegaron a R2, pero el servidor puede seguir procesando. F17 debe transicionar a `analyzing` solo cuando recibe la respuesta del endpoint, no cuando `uploadProgress === 100` |
