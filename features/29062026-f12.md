## Tipo `OutfitDescription`

Este tipo es el contrato entre GPT-4o-mini (F12) y Minimax M3 (F13). Su diseño es deliberado: campos concretos y descriptivos, sin subjetividad.

```ts
export interface Garment {
  type:     string    // 'top' | 'bottom' | 'dress' | 'jacket' | 'outerwear' | 'footwear' | etc.
  color:    string    // nombre del color en inglés: 'navy blue', 'off-white', etc.
  pattern:  string    // 'solid' | 'stripes' | 'plaid' | 'floral' | 'geometric' | 'animal print' | etc.
  fit:      string    // 'oversized' | 'relaxed' | 'regular' | 'slim' | 'fitted' | 'tailored' | etc.
  texture:  string    // 'cotton' | 'denim' | 'silk' | 'knit' | 'leather' | 'linen' | etc.
  length?:  string    // 'cropped' | 'midi' | 'maxi' | 'full-length' — solo para prendas con largo relevante
}

export interface OutfitDescription {
  garments:          Garment[]
  accessories:       string[]    // ['leather belt', 'white sneakers', 'baseball cap', ...]
  dominantColors:    string[]    // los 2-4 colores que dominan visualmente el outfit
  colorTemperature:  'warm' | 'cool' | 'neutral' | 'mixed'
  patterns:          string[]    // todos los patrones detectados en el outfit
  layering:          boolean     // hay capas visibles (chaqueta sobre camiseta, etc.)
  overallStyle:      string      // mejor aproximación al estilo: 'casual' | 'streetwear' | 'formal' | etc.
  silhouetteNotes:   string      // cómo el outfit interactúa con la silueta percibida — descripción objetiva
  colorInteraction:  string      // cómo los colores del outfit interactúan entre sí (descripción)
  fitObservations:   string      // observaciones generales sobre el ajuste y proporciones del outfit
  confidence:        'high' | 'medium' | 'low'  // qué tan clara es la imagen para el análisis
}
```

**Por qué `silhouetteNotes`, `colorInteraction` y `fitObservations` son strings y no enums**: son observaciones descriptivas que no pueden reducirse a un enum sin perder información valiosa. Minimax M3 los lee como texto y los interpreta con su conocimiento de moda. GPT-4o-mini es bueno generando descripciones concisas y precisas en inglés.

**Por qué `confidence`**: si la imagen es borrosa, está mal iluminada o el outfit apenas se ve, GPT lo indica con `'low'`. Minimax M3 puede entonces ponderar su análisis con más cautela o el sistema puede advertir al usuario que suba una foto más clara.

### T12.1 — `src/lib/outfitDescription.ts` — Tipos de descripción visual

Crear el archivo con la interfaz `Garment` y `OutfitDescription` tal como están definidas en el apartado "Tipo `OutfitDescription`" de este documento.

Adicionalmente, la clase de error de extracción:

```ts
export type ExtractionErrorCode =
  | 'INVALID_RESPONSE'   // el JSON no cumple el schema mínimo
  | 'OPENAI_FAILED'      // la llamada a OpenAI falló
  | 'EMPTY_RESPONSE'     // OpenAI retornó una respuesta vacía

export class ExtractionError extends Error {
  constructor(public readonly code: ExtractionErrorCode, message?: string) {
    super(message ?? code)
  }
}
```

**Schema mínimo para validación**: un `OutfitDescription` es válido si:
- `garments` es un array con al menos 1 elemento
- Cada garment tiene al menos `type`, `color`, `pattern`, `fit`, `texture` como strings no vacíos
- `dominantColors` es un array con al menos 1 string
- `colorTemperature` es uno de los 4 valores literales
- `confidence` es uno de los 3 valores literales

Función validadora:

```ts
export function isValidOutfitDescription(raw: unknown): raw is OutfitDescription
```

La función no valida los strings internos (no verifica que `fit` sea uno de los valores conocidos) — GPT produce vocabulario descriptivo variado y eso es deseable. Solo valida la estructura.

**Criterio de aceptación**: `isValidOutfitDescription` retorna `false` para `{}`, `null`, y para un objeto sin el campo `garments`. Retorna `true` para un objeto con todos los campos requeridos.

### T12.3 — `src/lib/prompts/gpt4oMini.ts` — Prompt de extracción visual

**System prompt — estructura obligatoria**:

El system prompt instruye a GPT-4o-mini a comportarse como un observador objetivo y detallado, sin emitir juicios.

1. **Rol**: observador experto en descripción visual de moda. Detecta y describe con precisión todo lo que ve en la imagen. No evalúa, no puntúa, no recomienda.

2. **Qué describir** — instrucciones explícitas para cada campo del `OutfitDescription`:
   - Para cada prenda visible: tipo, color exacto, patrón, corte/fit, textura percibida, largo si aplica.
   - Accesorios y calzado visibles.
   - Los 2-4 colores dominantes del outfit en su conjunto.
   - La temperatura de color del conjunto (warm/cool/neutral/mixed).
   - Todos los patrones presentes.
   - Si hay layering visible.
   - La mejor aproximación al estilo general del outfit.
   - Una nota de silueta: cómo el outfit interactúa con la forma percibida del cuerpo.
   - Una descripción de la interacción de colores.
   - Observaciones sobre ajuste y proporciones generales.
   - El nivel de confianza basado en la claridad de la imagen.

3. **Restricciones explícitas**: NO incluir scores numéricos, NO incluir palabras como "good", "bad", "beautiful", "ugly", "works", "doesn't work" — solo descripción neutral. Si no puede determinar un campo, usar `"not visible"` como valor.

4. **Schema JSON exacto de respuesta**: el JSON del `OutfitDescription` completo. El modelo retorna SOLO este JSON.

**Función de contexto del usuario** — construye el texto adicional que orienta a GPT con los datos físicos del usuario:

```ts
interface ExtractionContext {
  height:    number | null
  build:     { upper: string | null; lower: string | null }
  skinColor: string | null   // label en inglés
}

export function buildExtractionContext(ctx: ExtractionContext): string
```

Este contexto no le dice a GPT cómo analizar — solo le ayuda a describir la silueta en relación a las medidas reales del usuario. Ejemplo: "User height: 170cm, build: upper athletic / lower regular. Use this to describe how the outfit proportions relate to the body."

**Por qué el contexto del usuario va en el user message y no en el system prompt**: el system prompt define el comportamiento genérico del extractor. El contexto específico del usuario es datos de la petición, no configuración del rol.

**Criterio de aceptación**: el system prompt no contiene ninguna instrucción de scoring. `buildExtractionContext({ height: 175, build: { upper: 'Athletic', lower: 'Regular' }, skinColor: 'Medium' })` retorna un string de contexto legible en inglés.

### T12.4 — `src/lib/extractOutfitDescription.ts` — Función de extracción

```ts
import type { OutfitDescription, ExtractionErrorCode } from '@/lib/outfitDescription'
import { ExtractionError, isValidOutfitDescription } from '@/lib/outfitDescription'
import { openai } from '@/lib/openai'
import { env } from '@/lib/env'
import { GPT4O_MINI_SYSTEM_PROMPT, buildExtractionContext } from '@/lib/prompts/gpt4oMini'
import type { FormParams } from '@/components/organisms/HomeForm'

export interface ExtractionInput {
  objectKey:  string
  formParams: FormParams
}

export async function extractOutfitDescription(
  input: ExtractionInput,
): Promise<OutfitDescription>
```

**Flujo interno**:
1. Construir la URL pública de la imagen: `${env.r2.publicUrl}/${input.objectKey}`
2. Construir el texto de contexto del usuario con `buildExtractionContext`
3. Llamar a `openai.chat.completions.create`:
   - `model: 'gpt-4o-mini'`
   - `temperature: 0.1`
   - `max_tokens: 800`
   - `response_format: { type: 'json_object' }`
   - System message: `GPT4O_MINI_SYSTEM_PROMPT`
   - User message: imagen como `image_url` + texto de contexto del usuario
4. Verificar que `choices[0]?.message?.content` existe — si no, lanzar `ExtractionError('EMPTY_RESPONSE')`
5. Parsear JSON: si falla, lanzar `ExtractionError('OPENAI_FAILED')`
6. Validar con `isValidOutfitDescription`: si falla, lanzar `ExtractionError('INVALID_RESPONSE')`
7. Retornar el `OutfitDescription` validado

**Por qué `extractOutfitDescription` PUEDE lanzar (a diferencia de `analyzeWithMinimax` en F13 que es fail-open)**: si GPT-4o-mini no puede describir el outfit, no hay nada que pasarle a Minimax. El error debe propagarse al endpoint para que informe al usuario. Es una falla crítica del pipeline, no una falla de un componente opcional.

**Criterio de aceptación**: con una imagen real en R2, la función retorna un `OutfitDescription` completo. Con una URL inválida (imagen que no existe), lanza `ExtractionError`. El `OutfitDescription` retornado no contiene ningún número de score.

## Riesgos

| Riesgo | Mitigación |
|---|---|
| GPT-4o-mini incluye juicios de valor en los campos de texto a pesar del prompt | Revisar varias respuestas reales y ajustar el prompt en el system message. Agregar ejemplos explícitos de lenguaje neutral vs. lenguaje de juicio |
| El JSON retornado tiene campos extra no definidos en `OutfitDescription` | El validador solo verifica la presencia de campos requeridos, no la ausencia de extras. Los extras son ignorados en TypeScript con la asignación tipada. No es un riesgo crítico |
| `confidence: 'low'` en imágenes oscuras o borrosas sin aviso al usuario | F13 y F17 deben manejar este valor: si `confidence === 'low'`, la UI puede mostrar un aviso antes de los resultados. Documentar como requisito para F16 |
| La imagen HEIC no es procesable por GPT-4o-mini | GPT-4o-mini no soporta HEIC nativamente. F18 (Apple compatibility) convierte HEIC a JPEG en el cliente antes de subir. Documentar aquí como dependencia de F18 |
| `max_tokens: 800` insuficiente para outfits muy complejos (muchas prendas) | En ese caso GPT trunca la respuesta y el JSON es inválido. Si `isValidOutfitDescription` falla con frecuencia en outfits complejos, subir `max_tokens` a 1200 |
